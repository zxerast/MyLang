#  Начало программы

Программа на  вход получает до 4х аргументов: 

./lang <Имя файла с непосредственным кодом>.lang -o <Имя выходного файла> --dump-tokens --dump-ast

Обязательным является только первый, остальные опциональны. В случае отсутствия флага -o имя выходного
файла будет совпадать с изначальным. (main.cpp :: 110)

-dump--tokens - Вывод всех полученных токенов после токенизации (main :: 68)
-dump--ast - Вывод всех узлов AST из вектора деклараций полученного после парсинга (main.cpp :: 86)

После проверки на существование файла, создаётся поток чтения code_file и привязывается к данному файлу,
создаётся поток buf для перевода данных из потока чтения в тип string (main.cpp :: 53-60)

В итоге весь код преобразовывается в одну строку типа string, которая отправляется в лексический анализатор

#  Лексический анализатор

На вход поступает 2 string массива: код полученный на прошлом этапе и имя самого файла для сообщений об
ошибках. Точка входа - tokenize, в ней создаётся вектор токенов и инициализируются стартовые номера строки 
и столбца для сообщений об ошибках (Lexer.cpp :: 118)

Токен представляется в виде структуры Token имеющий в себе 4 поля:

1) type - enum тип TokenType который представляет собой один из 53 зарезервированных типов символов и имён
используемых в коде (Tokens.hpp :: 7-61)
2) lexeme - непосредственно сам символ типа string
3) line - номер линии 
4) column - номер столбца

Токен создаётся с помощью конструктора принимающего 2 первых поля как аргументы.

Цикл проходится по всей строке кода и разбирает символ. Каждый подходящий символ создаётся конструктором
напрямую в векторе токенов с помощью emplace_back(). Некоторые токены требующие несколько символов (++, --)
при проверке захватывают соответствующее количество следующих символов. В случае встречи символа "//"
последующие символы пропускаются без создания токенов до следующего "\n". В случае встречи "\n" номер
строки увеличивается на 1, а номер столбца обнуляется. В случае встречи '"' создаётся временный буфер для 
записи строк, в него записываются все найденные символы до встречи следующей '"', escape последовательности
из C также обрабатываются и добавляются, в конце буфер добавляется в общий вектор токенов как аргумент с 
типом StringLit. (Lexer.cpp :: 244) 

Если в коде не был встречен зарезервированный символ, значит это либо цифра, либо буква (Lexer.cpp :: 271) 
Буквы обрабатываются аналогично случаю со строками в кавычках и собираются в слова, затем через
typeIdentifier определяется сущность которая стоит за этим словом, один из 53 зарезервироанных ключевых
слов TokenType, и с определённым типом также создаётся в конце вектора токенов.

Строки состоящие исколючительно из чисел обрабатывают случай с наличием или отсутствием точки и 
добавляются в общий вектор с типом Number и выставленным флагом isFloat.

Особые случаи: случай 5. - разрешён и будет представлен в формате float 5.0. Случай .5 - запрещён из-за
конфликта с семантикой обращения к полю структур obj.x, .5 вернёт ошибку "." unexpected token.

Последний добавленный токен получает текущий номер строки и столбца в конце цикла. Последний токен всегда 
End. Все ошибки возвращаются в формате: <Имя файла>:<Строка>:<Столбец>:<Текст ошибки> через std::unexpected

Функция возвращает вектор токенов типа Token.

#  Парсинг

Точка входа парсинга - parse . На вход поступает вектор токенов сформированный лексером
и имя файла для сообщений об ошибках. Создаётся объект head типа Parser с переданными в него токенами и
именем файла. (Parser.cpp :: 1454) 

Объявляется вектор инструкций в котором последовательно будут лежать AST структуры определённые парсером.
Базовый тип узла ASTNode обладает только номером строки и столбца которые передаются из текущего токена. 
(Ast.hpp :: 8)

Каждая инструкция языка обладает уникальной AST структурой, но в общем их можно поделить на две 
категории: 

1) Statements (Stmt) - инструкции управляющие потоком исполнения и состояния программы, не проводят
вычисления и не возвращают значения. 

2) Expression (Expr) - инструкции выполняющие прямое вычисление результата по заданным параметрам, всегда
возвращает значение. Все expr узлы обладают полем resolvedType - тип результата который определяется
семантическим анализатором.(См. главу "Семантический анализатор") 

Парсинг начинается с разбора всех import, так как по спецификации они должны идти вначале файла. Затем идёт 
разбор так называемых Top Declarations, или инструкций из глобальной области видимости. К
ним относятся: глобальные переменные, объявления функций, структур, классов, пространств имён, подключение
модулей, константы, синонимы типов. Все обявления в глобальной области видимости видны сразу, что позволяет
вызывать в main функции объявленные позже, или инициализировать переменные структурами и классами также
описанными позже. Каждая инструкция определяется её токеном и вызывается определённый метод Parse для
парсинга соответствующей инструкции:   (Parser.cpp :: 752-792)

## import

Подключение сторонних модулей различает два случая:

1) import module.lang - импорт стороннего модуля на таком же языке
2) import <stdio.h> - импорт C библиотеки или C модуля

Структура имеет только 1 поле имени модуля и флаг isC помечающий библиотеку как стороннюю (Ast.hpp :: 195)

## export

Экспорт используется для пометки деклараций, доступных вне текущего модуля:

export declaration

Парсер считывает ключевое слово export, после чего рекурсивно вызывает parseTopDecl() для разбора
внутренней декларации.

В результате создаётся узел ExportDecl, содержащий:

* decl — вложенная декларация

## struct

Структуры парсятся в виде:

struct Point{
    int x = 4;
    float y;
}

Структуры обладают именем и вектором полей где каждое поле обладает типом, именем и значением по умолчанию.
(для x значение по умолчанию 4, для y - 0.0, тип expr). Парсер сперва съест ключевое слово, имя и 
левую скобку структуры и пока не встретит закрывающую, будет вызывать parseTypeName() для стандартных
случаев объявления int x, а если встретит "=" после объявления то дополнительно вызовется 
parseEquasion() для парсинга инициализации (См. пункт "Парсинг expression"), и полученное значение ставится
значением по умолчанию у структуры. (Parser.cpp :: 383-432)

## Type name

Функция parseTypeName при вызове сперва проверяет является ли текущий токен именем типа (зарезервированный 
typename или пользовательский struct iden) и создаёт объект TypeName с соответствующим именем типа. 

Если мы нашли после имени токен "::" то это означает обращение к типу из пространства имён и имя типа 
перезаписывается в виде "namespace::TypeName". Работает и для вложенных пространств имён в цикле.

Если после имени типа стоит "[" значит мы имеем дело с массивным типом данных, а это либо:

1) type[] - динамический массив
2) type[const expr] - статический массив
3) type[][]... - многомерный динамический массив
4) type[const expr][const expr]... - многомерный статический массив
5) type[][const expr]... - смешанный тип

В многомерном массиве первое значение от типа - это размер элемента, второе - кол-во элементов.

int[2][3] arr; - массив из 3х массивов размера 2

Каждое новое открытие [] порождает новый суффикс у типа.
Если сразу после '[' стоит ']' значит это динамический массив и в тип заносится один суффикс с флагом 
isDynamic.
Если после '[' стоит выражение то это выражение парсится с помощью parseEquasion и сохраняется в поле 
суффикса типа, в коне суффикс добавляется к типу.
Так продолжается до тех пор пока у нас в типе будет незакрытая "[", в конце функция возвращает собранный тип.

## Class

Классы парсятся в виде:

class ClassName{
    ClassName(){}    //  Конструктор
    void Method(float x){}  //  Метод
    int s;  //  Поле
    ~ClassName()    //  Деструктор
}

Структура класса хранит в себе имя, вектор полей (идентичный как у структур), вектор методов типа 
аналогичного функциям, а также отдельные указатели на конструктор и деструктор. (Ast.hpp :: 166-171)

Тела методов, конструктора и деструктора парсятся через отдельный вызов parseBlock(), Деструктор всегда
имеет тип возврата void.

## Function Declaration

Объявление функций парсится в формате:

type name(type1 arg1, type2 arg2, ...){
// тело функции
}

Парсер начинает с вызова parseTypeName() для определения возвращаемого типа функции. Далее ожидается
идентификатор имени функции и открывающая круглая скобка. После этого начинается разбор списка параметров.

Каждый параметр представляет собой пару type name, где тип снова парсится через parseTypeName(), а имя
берётся из токена Iden. Параметры разделяются запятыми и добавляются в вектор params структуры FuncDecl.

После закрывающей скобки ожидается тело функции, заключённое в фигурные скобки. Тело функции парсится
через вызов parseBlock(), который возвращает узел типа Block содержащий список инструкций.

В результате формируется AST узел FuncDecl, содержащий:

* returnType — строковое представление возвращаемого типа
* name — имя функции
* params — список параметров
* body — тело функции в виде блока инструкций

(Parser.cpp :: 309-380)

## Variable Declaration

Объявление переменных имеет следующий общий вид:

type name;
type name = expression;
const type name = expression;
auto name = expression;

Парсер сначала проверяет наличие модификатора const, после чего проверяет ключевое слово auto. В случае
auto тип переменной не задаётся явно и будет определён на этапе семантического анализа.

Если auto отсутствует, вызывается parseTypeName() для получения строкового представления типа. Далее
ожидается идентификатор имени переменной.

Если после имени следует символ '=', вызывается parseEquasion() для разбора выражения инициализации.
Полученное выражение сохраняется в поле init.

Объявление обязательно должно заканчиваться символом ';', иначе возвращается ошибка.

В результате формируется AST узел VarDecl, содержащий:

* typeName — тип переменной (если не auto)
* name — имя переменной
* init — выражение инициализации (или nullptr)
* isConst — флаг константы
* isAuto — флаг автоматического вывода типа

(Parser.cpp :: 798-848)

## Block

Блок инструкций представляет собой последовательность Statement, заключённых в фигурные скобки:

{
stmt1;
stmt2;
...
}

Парсер создаёт узел Block и последовательно вызывает parseStatement() до тех пор, пока не встретит
закрывающую фигурную скобку.

Каждая успешно распознанная инструкция добавляется в вектор statements. В случае ошибки разбор блока
прерывается и ошибка пробрасывается выше.

Закрывающая скобка '}' обязательна, иначе возвращается ошибка.

В результате формируется AST узел Block, содержащий:

* statements — вектор инструкций

(Parser.cpp :: 850-872)

## If Statement

Условный оператор имеет следующий вид:

if (condition){
// then
} else {
// else
}

Парсер сначала считывает ключевое слово if, затем ожидает условие в круглых скобках, которое разбирается
через parseEquasion().

После условия обязательно ожидается блок then, заключённый в фигурные скобки и разбираемый через
parseBlock().

Опционально может следовать ветка else. Она может быть представлена либо блоком, либо вложенным if
(else if), который парсится рекурсивным вызовом parseIf().

В результате формируется AST узел If, содержащий:

* condition — выражение условия
* thenBranch — блок then
* elseBranch — либо блок else, либо другой If, либо nullptr

(Parser.cpp :: 157-207)

## While Statement

Цикл while имеет следующий вид:

while (condition){
// тело цикла
}

Парсер считывает ключевое слово while, затем ожидает условие в круглых скобках, которое разбирается
через parseEquasion().

После условия обязательно ожидается тело цикла в фигурных скобках, которое разбирается через parseBlock().

В результате формируется AST узел While, содержащий:

* condition — выражение условия
* body — тело цикла

(Parser.cpp :: 218-253)

## Return / Break / Continue

Инструкции управления потоком исполнения:

return expression;
break;
continue;

Для return выражение является опциональным. Если после ключевого слова отсутствует ';', вызывается
parseEquasion() для разбора возвращаемого значения.

Инструкции break и continue не принимают аргументов и должны заканчиваться символом ';'.

В результате формируются соответствующие AST узлы:

* Return (value может быть nullptr)
* Break
* Continue

(Parser.cpp :: 255-307)

## Expression Parsing

Парсинг выражений реализован через иерархию функций, каждая из которых отвечает за свой уровень
приоритета операций. Точка входа — parseEquasion(), которая делегирует разбор в parseLogicOr().

Далее разбор происходит по цепочке:

parseLogicOr → parseLogicAnd → parseEquality → parseComparison → parseAddSub → parseMulDiv → parseUnary → parsePostfix → parsePrimary

Каждый уровень отвечает за конкретный набор операторов и обеспечивает корректный приоритет. Например,
умножение и деление обрабатываются раньше сложения и вычитания за счёт того, что parseAddSub() вызывает
parseMulDiv(), а не наоборот.

На каждом уровне используется цикл, который позволяет обрабатывать левую ассоциативность операторов,
создавая цепочку узлов Binary.

В результате формируется AST дерево выражения, где каждый узел отражает конкретную операцию или значение.

(Parser.cpp :: 874-1457)

## Binary Operations

Бинарные операции (такие как +, -, *, /, ==, && и др.) парсятся на соответствующих уровнях приоритета.

После разбора левого операнда, парсер проверяет наличие оператора текущего уровня. Если оператор найден,
он сохраняется в виде значения enum Operand, после чего разбирается правый операнд.

Для каждой операции создаётся узел Binary, содержащий:

* op — тип операции
* left — левый операнд
* right — правый операнд

Полученный узел становится новым левым операндом, что позволяет корректно обрабатывать цепочки операций:

a + b + c → ((a + b) + c)

## Unary Operations

Унарные операции включают:

* отрицание (-x)
* логическое отрицание (!x) //  Только для bool операндов
* унарный плюс (+x)

Парсер проверяет наличие соответствующего токена и при его обнаружении рекурсивно вызывает parseUnary()
для разбора операнда.

Для каждой операции создаётся узел Unary, содержащий:

* op — тип операции
* operand — выражение

Постфиксные операции (x++, x--) обрабатываются отдельно на этапе parsePostfix().

## Postfix Expressions

Постфиксные выражения включают:

* доступ к полю: obj.field
* доступ по индексу: arr[i]
* вызов функции: func(a, b)
* постфиксный инкремент/декремент: x++, x--

Парсер сначала получает базовое выражение через parsePrimary(), после чего в цикле проверяет наличие
постфиксных операций.

Каждая операция оборачивает текущее выражение в новый AST узел:

* FieldAccess — для доступа к полю
* ArrayAccess — для индексации
* FuncCall — для вызова функции
* Unary — для ++ и --

## Primary Expressions

Базовые выражения (листья AST) включают:

1. Числовые литералы
   Строка преобразуется в число через std::stod, при наличии точки устанавливается флаг isFloat.

2. Строковые литералы
   Сохраняются как есть в узле String.

3. Символьные литералы
   Строки длины 1

4. NULL литерал
   Значение несозданного экземпляра класса

5. Cast литерал
   Изначальный тип -> целевой тип

6. Булевы литералы
   true / false преобразуются в соответствующее значение.

7. Идентификаторы
   Создаётся узел Identifier, содержащий имя переменной или функции.

8. Группировка (expr)
   Выражение в скобках рекурсивно разбирается через parseEquasion().

9. Литералы массивов
   [expr1, expr2, ...] → ArrayLiteral

10. Литералы структур
   Type { field: value, ... } → StructLiteral

11. Доступ к пространству имён
   namespace::member → NamespaceAccess


## Function Call

Вызов функции определяется при встрече конструкции:

callee(expr1, expr2, ...)

Парсер создаёт узел FuncCall, где:

* callee — выражение, представляющее вызываемый объект
* args — список аргументов

Аргументы парсятся через parseEquasion() и разделяются запятыми.

Вызов функции имеет более высокий приоритет, чем бинарные операции, поэтому обрабатывается на этапе
parsePostfix().


## Cast Expression

Приведение типов реализовано через синтаксис:

cast<type>(expression)

Парсер ожидает:

* ключевое слово cast
* тип в угловых скобках
* выражение в круглых скобках

После успешного разбора создаётся узел CastExpr, содержащий:

* targetType — целевой тип
* value — выражение для приведения

## Assignment

Присваивание обрабатывается на уровне parseStatement(), а не внутри выражений, что делает его
отдельным видом инструкции:

lvalue = expression;

Парсер сначала разбирает левую часть через parseEquasion(). После этого проверяется наличие одного из 
символов присваивания.
Если он присутствует, вызывается parseEquasion() для разбора правой части.

Важно, что левая часть должна представлять собой допустимый lvalue (переменная, поле структуры,
элемент массива), однако эта проверка откладывается на этап семантического анализа.

После разбора ожидается символ ';'. В результате создаётся узел Assign, содержащий:

* target — левая часть
* value — правая часть

## Type Alias

Синонимы типов задаются конструкцией:

type Alias = OriginalType;

Парсер считывает ключевое слово type, затем имя псевдонима и символ '='. После этого вызывается
parseTypeName() для разбора исходного типа.

Объявление должно завершаться символом ';'. В результате создаётся узел TypeAlias, содержащий:

* alias — имя нового типа
* original — строковое представление исходного типа

## Namespace

Пространства имён позволяют группировать декларации:

namespace Name {
// декларации
}

Парсер считывает ключевое слово namespace и имя, после чего ожидает блок в фигурных скобках.

Внутри блока вызывается parseTopDecl() для каждой декларации верхнего уровня, что позволяет
вкладывать структуры, функции, классы и другие элементы.

В результате создаётся узел NamespaceDecl, содержащий:

* name — имя пространства имён
* decls — список вложенных деклараций


## Array Literal

Литералы массивов задаются в виде:

[expr1, expr2, ...]

Парсер создаёт узел ArrayLiteral и последовательно разбирает элементы через parseEquasion(),
разделённые запятыми.

Пустой массив [] также допустим. После завершения списка обязательно ожидается символ ']'.

В результате формируется узел:

* elements — список выражений

## Struct Literal

Литералы структур задаются следующим образом:

Type {field1: value1, field2: value2}

Парсер сначала считывает имя типа, затем при встрече '{' начинает разбор списка инициализации полей.

Каждое поле задаётся как name : expression. Поля разделяются запятыми.

В результате создаётся узел StructLiteral, содержащий:

* name — имя структуры
* fields — список пар (имя поля, значение)

## Namespace Access

Доступ к элементам пространства имён реализован через:

namespace::member

Парсер при встрече '::' создаёт узел NamespaceAccess, содержащий:

* nameSpace — имя пространства имён
* member — имя элемента

##   Конец парсинга

После завершения разбора всех деклараций функция parse() возвращает вектор указателей на узлы AST.

Цикл продолжается до тех пор, пока не будет достигнут токен End, который всегда добавляется лексером
в конец входного потока.

В случае ошибки на любом этапе парсинга возвращается std::unexpected с сообщением об ошибке, содержащим
имя файла, строку и позицию.

Таким образом, результатом работы парсера является вектор AST узлов деклараций программы, полностью
готовый к семантическому анализу.

# Семантический анализатор

После этапа парсинга в main создаётся объект анализатора и вызывается точка входа analyze, на которую 
подаются собранные AST узлы в виде структуры с новым именем Program и путь к текущему файлу для ошибок и 
отслеживания модулей.

## Первый обход - анализ импортов

Первый обход семантического анализатора посвящён исключительно импортам и запускается ещё до 
предобъявления остальных top-level деклараций. 

В начале analyzeModule() анализатор очищает накопленные ошибки и список невалидных деклараций, 
канонизирует путь текущего файла, добавляет его в множество importedFiles для защиты от повторного 
захода в уже разбираемый модуль, очищает текущее пространство имён и регистрирует встроенные 
функции в глобальном scope. После этого вводится флаг seenNonImport, и цикл последовательно проходит 
program->decls сверху вниз: пока встречаются ImportDecl, они считаются допустимыми, но как только 
найдено любое другое top-level объявление, ставится флаг, и каждый следующий import уже выдаёт ошибку. 

Каждый найденный импорт передаётся в processImport(), где сначала разделяются два случая: import <...> 
сразу обрабатывается как C-импорт через processCImport(), а обычный import module.lang разбирается в этой же функции.

##  Импорт модулей .lang

Файл модуля разрешается относительно директории текущего файла, проверяется на существование и 
приводится к каноническому абсолютному пути. Если такой путь уже лежит в importedFiles, импорт молча 
пропускается, что разрывает циклы вида A -> B -> A.

Иначе путь заносится в множество импортируемых файлов, файл читается, повторно проходит стадии 
токенизации и парсинга, после чего для импортируемого AST создаётся отдельный SemanticAnalyzer со своей 
таблицей символов и внутренним состоянием, для предотварщения конфлитов имён в одной общей таблице. 

Анализатору модуля передаётся уже накопленное множество импортированных файлов, чтобы рекурсивный анализ 
дочерних модулей также видел текущую цепочку импортов и не уходил в бесконечную рекурсию. После 
успешного analyzeModule() все собранные подмодулем пути возвращаются в родительское множество 
importedFiles, из таблицы символов дочернего анализатора в текущий модуль копируются только 
экспортированные сущности через importExportedSymbolsFrom() — встроенные функции и 
любые символы без флага isExported специально отбрасываются, а AST импортированного модуля 
добавляется в ownerProgram->imports, чтобы основной модуль хранил дерево всех реально подключённых 
зависимостей. 

##  Импорт модулей C

Ветка import <header.h> не читает .lang файл и не запускает рекурсивный разбор модулей. Вместо этого
processCImport() создаёт виртуальный C файл с одной строкой #include <header.h> и передаёт его в
libclang через механизм unsaved file, чтобы не создавать временный файл на диске. Для парсинга
явно задаются аргументы -x c, что заставляет clang всегда интерпретировать вход как C код.

После этого clang_parseTranslationUnit2() строит AST заголовка без тел функций. 
При успешном разборе берётся корневой узел C AST и обходится функцией clang_visitChildren(). Из всех 
узлов дерева выбираются только FunctionDecl, причём дополнительная проверка isFromHeader() отбрасывает 
объявления, пришедшие из вложенных зависимостей заголовка: в таблицу символов попадают только функции 
именно из того header-файла, который был указан в import.

Для каждой подходящей C-функции вызывается makeCFuncSymbol(). Эта функция считывает имя функции, тип
возврата и параметры из libclang, преобразует C типы во внутреннее представление языка и создаёт
обычный символ функции с флагом инициализации. Если типы функции не удалось корректно сопоставить
внутренней системе типов, такая функция пропускается. Иначе символ сразу регистрируется в текущей
таблице символов, и после завершения обхода импортированный C API становится доступен остальной
семантике точно так же, как и обычные функции, объявленные в языке.

##  Второй проход - predeclareTopLevel

Второй проход запускается сразу после полного разбора импортов и до collectTopLevel(). Его задача —
заранее собрать имена всех деклараций верхнего уровня и положить в таблицу символов заглушки, ещё без
полностью разобранных сигнатур и тел. Благодаря этому порядок объявлений внутри файла не влияет на
видимость имён: например, функция main может вызвать функцию add, даже если add объявлена ниже по коду.

Точка входа прохода — predeclareTopLevel(). Функция получает вектор деклараций и последовательно
обходит его. Если встречается ExportDecl, анализатор не создаёт отдельный символ export, а рекурсивно
вызывает predeclareTopLevel() для вложенной декларации. То есть export на этом этапе работает как
обёртка над обычным top-level объявлением, а само имя всё равно регистрируется по правилам своего
реального типа.

Для FuncDecl создаётся символ с именем функции, kind = Function и пустым FuncInfo. Сигнатура, тип
возврата, параметры и проверки параметров здесь ещё не заполняются: они будут разобраны позже в
collectTopLevel(). На втором проходе важно только занять имя функции в текущем scope, чтобы последующие
объявления и тела функций могли на него ссылаться.

Для StructDecl и ClassDecl анализатор сразу собирает qualifiedName через appendQualifiedName() с учётом
текущего namespace. После этого создаётся символ с kind = Struct или kind = Class, создаётся пустая
информация StructInfo или ClassInfo, а также внутренний Type с TypeKind::Struct или TypeKind::Class.
В type->name записывается полное квалифицированное имя, чтобы вложенные пространства имён не смешивали
одноимённые типы между собой.

Для TypeAlias также создаётся заглушка символа, но его type пока остаётся nullptr. Вместо немедленного
разрешения типа в aliasTarget сохраняется исходный TypeName, а флаг isResolvingAlias выставляется в
false. Это нужно для ленивого разрешения alias: если синоним типа используется раньше, чем полностью
разобран его оригинальный тип, анализатор всё равно уже знает само имя alias и сможет разрешить его
позже.

NamespaceDecl обрабатывается отдельно, потому что namespace может быть объявлен несколько раз и должен
объединяться в одну область видимости. Для него создаётся символ kind = Namespace с отдельным
namespaceScope, а declareTopLevelSymbol() вызывается с флагом allowNamespaceMerge = true. Если namespace
с таким именем уже существует, новый символ не считается ошибкой, а получает ссылку на уже существующий
scope. Затем анализатор делает pushScope() в область namespace, сохраняет предыдущий currentNamespace,
добавляет к нему имя текущего namespace и рекурсивно вызывает predeclareTopLevel() для внутренних
деклараций. После обхода вложенного блока currentNamespace возвращается назад, а scope закрывается через
exitScope().

Глобальные VarDecl на этом проходе тоже регистрируются, но только как имена переменных. Для каждой
переменной из var->vars создаётся символ kind = Variable, переносится флаг const, isInitialized ставится
в false, а type остаётся nullptr. Реальный тип, auto, инициализация и проверка выражения будут
обработаны позднее, когда семантический анализ дойдёт до collectTopLevel() и анализа тел.

Все созданные заглушки проходят через declareTopLevelSymbol(). Эта функция проверяет текущий scope на
дубли имени. Если имя уже занято и это не допустимое слияние namespace, декларация добавляется в
invalidTopLevelDecls, а анализатор выдаёт ошибку duplicate top-level name с указанием, какой символ уже
объявлен в этой области. Если дубля нет, символ спокойно добавляется в таблицу. Таким образом второй
проход формирует карту всех верхнеуровневых имён модуля и заранее отсекает конфликтующие объявления, не
переходя ещё к подробному анализу типов, сигнатур и выражений.

##  Третий проход - collectTopLevel

Третий проход запускается после predeclareTopLevel() и проходит по тем же top-level декларациям, но теперь
не просто занимает имена, а полноценно заполняет символы информацией о типах, сигнатурах, полях и
экспорте. В начале каждой итерации collectTopLevel() проверяет invalidTopLevelDecls: если декларация
уже была признана некорректной на этапе предобъявления из-за дубля имени, она пропускается и не
перезаписывает таблицу символов повторно.

Для FuncDecl создаётся новый Symbol с kind = Function. Анализатор проверяет дубли параметров через
checkDuplicateParams(), разрешает тип возврата через resolveTypeName() и при ошибке пишет unknown
return type. После этого создаётся FuncInfo, куда заносится returnType и список параметров. Каждый
параметр проходит через resolveDeclaredType(), поэтому здесь уже проверяется его объявленный тип,
const/auto и defaultValue в контексте параметра. Полученный ParamInfo добавляется в сигнатуру функции,
а сам AST узел получает ссылки resolvedSym и resolvedInfo. В конце символ заново объявляется через
table.declare(), тем самым заполненный символ заменяет заглушку из второго прохода.

Для StructDecl также создаётся новый символ, но теперь structInfo заполняется реальными полями. Сначала
собирается qualifiedName с учётом currentNamespace, затем checkDuplicateFields() проверяет повторяющиеся
имена полей внутри структуры. Каждое поле разрешает свой тип через resolveDeclaredType() в контексте
Field, сохраняет результат в field.resolvedType и добавляется в structInfo->fields вместе с именем,
типом, флагом const и defaultValue. После этого создаётся TypeKind::Struct с полным именем структуры и
символ регистрируется в таблице.

TypeAlias на третьем проходе не обязан создавать новый символ с нуля, потому что его имя уже должно быть
зарегистрировано в predeclareTopLevel(). collectTopLevel() сначала пытается найти alias в текущем scope,
и если по какой-то причине его нет или это не TypeAlias, создаёт символ заново и объявляет его. Основная
работа этого прохода для alias — сохранить original TypeName в aliasTarget. Сам исходный тип не
разворачивается немедленно, потому что алиасы разрешаются лениво при дальнейшем обращении к ним.

NamespaceDecl работает рекурсивно. Анализатор находит заранее объявленный символ namespace в текущем
scope, проверяет наличие namespaceScope, входит в него через pushScope(), сохраняет предыдущий
currentNamespace и добавляет к нему имя текущего namespace. Затем collectTopLevel() вызывается для
вложенных деклараций. После завершения вложенного обхода currentNamespace возвращается назад, а таблица
символов выходит из scope. Если namespace не был предобъявлен, это считается внутренней ошибкой, потому
что второй проход должен был заранее создать для него область видимости.

Для ClassDecl collectTopLevel() собирает почти всю метаинформацию класса. Сначала создаётся ClassInfo с
qualifiedName, затем проверяются дубли полей, методов и вложенных структур через checkDuplicateFields(),
checkDuplicateMethods() и checkDuplicateNestedStructs(). Поля класса обрабатываются так же, как поля
структуры: тип каждого поля разрешается, сохраняется в field.resolvedType и добавляется в
classInfo->fields.

Вложенные структуры класса обрабатываются раньше методов, потому что методы могут использовать их в
сигнатурах, например class A { struct S {...} S makeS() {...} }. Для каждой вложенной структуры
создаётся отдельный StructInfo с именем вида ClassName::StructName, проверяются дубли её полей, типы
полей разрешаются через resolveDeclaredType(), а готовая информация кладётся в
classInfo->nestedStructs по имени вложенной структуры.

Перед разбором методов анализатор временно сохраняет текущий currentClass и ставит currentClass равным
classInfo текущего класса. Для каждого метода создаётся FuncInfo: разрешается тип возврата, проверяются
дубли параметров, параметры проходят resolveDeclaredType(), а результат сохраняется в method->resolvedInfo
и classInfo->methods. Конструктор обрабатывается отдельно: его returnType всегда void, параметры
проверяются и записываются в constructor->resolvedInfo и classInfo->constructor. Деструктор также имеет
returnType void, но дополнительно проверяется, что у него нет параметров. После обработки методов,
конструктора и деструктора currentClass возвращается к предыдущему значению, для самого класса создаётся
TypeKind::Class с qualifiedName, и символ класса регистрируется в таблице.

ExportDecl на третьем проходе сначала рекурсивно вызывает collectTopLevel() для вложенной декларации,
чтобы она была собрана как обычный top-level элемент. После этого анализатор определяет имя
экспортируемой сущности: функцию, структуру, класс, alias или namespace можно пометить по одному имени,
а VarDecl может содержать несколько переменных, поэтому для него проход отдельно перебирает var->vars.
Все найденные символы получают isExported = true. Именно этот флаг затем используется при импорте .lang
модуля, когда importExportedSymbolsFrom() переносит наружу только разрешённые экспортом имена.

Для VarDecl третий проход заполняет символы глобальных переменных. Если переменная не auto, общий тип
объявления разрешается через resolveTypeName(); неизвестный тип вызывает ошибку unknown type, а void
запрещён как тип переменной. Затем для каждой переменной из var->vars ищется уже созданная заглушка в
текущем scope. Если её нет, символ создаётся и объявляется заново. AST узел переменной получает
resolvedSym, в символ переносятся isConst, isInitialized, isAuto и autoInit. Для auto тип пока остаётся
nullptr, потому что он будет выведен из инициализатора позже, но отсутствие инициализатора уже считается
ошибкой. Для обычной переменной в sym->type записывается объявленный тип.

Таким образом collectTopLevel() превращает карту имён из второго прохода в полноценную таблицу
top-level символов. После него анализатор уже знает сигнатуры функций и методов, поля структур и
классов, вложенные структуры, конструкторы, деструкторы, aliasTarget, глобальные переменные и флаги
экспорта. Следующий этап может анализировать тела функций и выражения, опираясь не на заглушки, а на
полностью собранную информацию о верхнем уровне модуля.

##  Четвёртый проход - analyzeStmt

После collectTopLevel() запускается четвёртый проход по program->decls. На этом этапе импорты уже
полностью обработаны, поэтому ImportDecl просто пропускаются, а все остальные top-level объявления
передаются в analyzeStmt(). В начале analyzeStmt() стоит общая защита от invalidTopLevelDecls: если
декларация была помечена невалидной на прошлых проходах, её тело или инициализатор повторно не
анализируются.

##  Инициализация переменных

Первой большой веткой analyzeStmt() является VarDecl. Здесь переменная уже может иметь имя в таблице 
символов после предыдущих проходов, но теперь анализатор разбирает реальную инициализацию, вывод auto, 
default значения и совместимость типов.

Сначала для объявления подготавливается общий declaredType. Если переменная не auto, анализатор вызывает
resolveTypeName() для typeName. Если тип не найден, добавляется ошибка unknown type. Если найденный тип
имеет TypeKind::Void, добавляется ошибка cannot declare variable with type 'void', потому что переменная
не может иметь тип void. Отдельно проверяется форма типа: если у typeName есть суффиксы и последний
суффикс является динамическим массивом, выставляется флаг isDynArrayDecl. Такой DynArray без явного
инициализатора считается уже инициализированным как пустой массив.

После этого проход идёт по каждой переменной из var->vars, потому что одно VarDecl может объявлять
несколько имён. Для имени переменной анализатор ищет символ в текущем scope через resolveCurrentScope().
Если символ найден и его kind = Variable, он считается уже объявленным, и анализатор будет только
обновлять поля существующей записи. Это обычный случай для глобальных переменных, которые были
зарегистрированы в collectTopLevel(). Если подходящего символа нет, создаётся новый Symbol с kind =
Variable, который позже будет добавлен в таблицу.

Затем в символ переносятся общие свойства объявления. isConst берётся из VarDecl, isInitialized сначала
ставится по наличию явного init у конкретной переменной. Если init отсутствует, но объявление является
динамическим массивом, isInitialized всё равно выставляется в true, потому что пустой динамический массив
считается допустимым начальным состоянием.

Для auto переменных тип выводится из инициализатора. Анализатор вызывает analyzeExpr(v->init), получает
тип выражения и записывает его в sym->type. Если тип получить не удалось, выдаётся ошибка auto
requires initializer to infer type for name. На практике это покрывает главный запрет auto без
инициализатора: без выражения анализатору не из чего вывести тип. После вывода типа intConstValue
сбрасывается. Если переменная одновременно const, имеет init, её тип известен и является целочисленным,
анализатор вызывает evalConstIntExpr() и, если выражение удалось вычислить на месте, сохраняет результат
в sym->intConstValue. Это нужно для дальнейших проверок константных размеров и других мест, где требуется
значение целочисленной константы.

Для не-auto переменных sym->type сразу получает declaredType. Если у const переменной нет явного
инициализатора, это ошибка const variable name requires an explicit initializer. Если переменная не
const, init отсутствует, а declaredType успешно найден и не является void, анализатор создаёт значение по
умолчанию через makeDefaultExprForType(), записывает его в v->init и помечает символ как
инициализированный. Поэтому обычные переменные с известным типом могут не иметь явного значения: оно
будет создано семантическим анализатором.

Дальше, если у переменной есть init, выражение инициализатора анализируется через analyzeExpr(v->init,
sym->type). Второй аргумент передаёт ожидаемый тип, чтобы литералы и составные выражения могли
анализироваться с учётом контекста переменной. После получения initType выполняется проверка
совместимости: если и initType, и sym->type известны, но isImplicitlyConvertible(initType, sym->type)
возвращает false, анализатор выдаёт ошибку cannot initialize name of type int with
string. Эта проверка запрещает, например, записать строку в int, но оставляет допустимые неявные
приведения вроде widening-преобразований чисел.

Для не-auto ветки intConstValue также сначала сбрасывается. Если переменная const, имеет init, её тип
известен и является целочисленным, анализатор снова пытается вычислить init как константное целое
выражение через evalConstIntExpr(). Успешный результат сохраняется в sym->intConstValue.

В конце обработки конкретной переменной анализатор проверяет флаг alreadyDeclared. Если символ не был
найден в текущем scope заранее, он объявляется через table.declare(); возможная ошибка объявления
добавляется в общий список ошибок. После этого AST узел переменной получает v->resolvedSym = sym. Таким
образом к концу этой ветки каждая переменная связана со своим символом, имеет флаги const/auto/init,
получает либо объявленный, либо выведенный тип, а её инициализатор уже проверен на существование,
корректность default-значения и допустимость приведения к типу переменной.

##  Присваивание

Следующая ветка analyzeStmt() обрабатывает Assign. На этом этапе присваивание уже распарсено отдельно от
обычных expression, поэтому семантический анализатор проверяет не только типы левой и правой части, но и
то, что цель вообще может быть изменена: является lvalue, не является const и допускает выбранный
оператор присваивания.

Сначала отдельно разбирается специальная форма присваивания в поле типа: TypeName.field = value или
Namespace::TypeName.field = value. Она используется не как запись в объект, а как переопределение
default-значения поля структуры или класса. Анализатор проверяет, что target является FieldAccess, а его
object — это Identifier или NamespaceAccess. Для Identifier символ ищется через table.resolve(), для
NamespaceAccess через resolveQualifiedSymbol(). Если найденный символ является Struct или Class,
анализатор ищет поле через findFieldInTypeSymbol().

Если такого поля нет, выдаётся ошибка struct/class has no field. Если поле найдено, но оно const,
переопределение default-значения запрещается ошибкой cannot redefine default of const field. Затем правая
часть анализируется через analyzeExpr(assign->value, field->type), где тип поля передаётся как ожидаемый
тип выражения. Если valueType существует, но не приводится к field->type через isImplicitlyConvertible(),
выдаётся ошибка cannot redefine default: expected ..., got .... При успехе FieldAccess получает
resolvedField, флаг isTypeDefaultFieldAccess и resolvedType поля, после чего обработка Assign сразу
заканчивается, потому что это не обычное присваивание в переменную.

Если цель не является переопределением default-поля типа, начинается обычная ветка присваивания.
Сначала анализируется левая часть через analyzeExpr(assign->target), затем правая часть через
analyzeExpr(assign->value, targetType). targetType передаётся во вторую функцию как контекст, чтобы
литералы и составные выражения справа могли сразу подстраиваться под ожидаемый тип цели.

Дальше проверяется lvalue. Функция isLvalue() разрешает присваивание только в переменную, поле,
элемент массива или переменную из namespace. Для Identifier дополнительно учитывается неявное поле
текущего класса: если имени нет в таблице, но currentClass содержит поле с таким именем, оно тоже
считается lvalue. Если цель не проходит эту проверку, например foo() = 5, 1 = x или (a + b) = 0,
анализатор выдаёт left-hand side of assignment is not an lvalue.

После lvalue идут проверки const. Сначала resolveTargetRoot() спускается по цепочке Identifier,
FieldAccess, ArrayAccess или NamespaceAccess до корневого символа цели. Это покрывает случаи вроде
arr[i] = x или obj.field = x: если корневой символ const, присваивание запрещается ошибкой cannot assign
to const. Затем отдельно проверяется Identifier, который мог быть разрешён как неявное self-поле класса;
если id->resolvedField существует и само поле const, выдаётся cannot assign to const field. Третья
проверка обрабатывает обычный FieldAccess obj.x = value, включая цепочки obj.a.b = value: анализатор
берёт resolvedType объекта, находит описание struct или class через resolveQualifiedSymbol(), ищет поле
по имени и запрещает запись, если поле помечено const.

Если левая часть является простым Identifier, анализатор дополнительно ищет его символ в таблице и
помечает sym->isInitialized = true. Это важно для дальнейших проверок использования переменной: после
успешного присваивания переменная считается инициализированной.

Финальная часть ветки работает только если удалось получить и targetType, и valueType. Для обычного
оператора = правая часть должна неявно приводиться к типу левой части. Проверка выполняется через
isImplicitlyConvertible(valueType, targetType). Если приведение невозможно, выдаётся type mismatch in
assignment: cannot assign actual to target.

Для compound-присваиваний логика другая, потому что выражение вида a += b семантически проверяется как
a = a + b. Анализатор заводит opOk и resultType. Для string += разрешена только конкатенация со string
или char, результатом считается исходный string. Для %= действует отдельное правило: оба операнда должны
быть целочисленными, иначе сразу выдаётся operator '%=' requires integer operands. Для остальных числовых
compound-операторов, если оба типа numeric, вычисляется общий числовой тип через commonType().

Если ни одно правило compound-оператора не подошло, анализатор выдаёт compound operator: incompatible
operand types. Если операция допустима, но resultType не может быть неявно приведён обратно к targetType,
выдаётся ошибка narrowing. Например, int8 += int64 даёт общий результат int64, и такой результат нельзя
безопасно записать обратно в int8. Таким образом compound-присваивание проверяет не только допустимость
самой операции, но и то, что результат операции помещается обратно в левую часть.

##  Условные операторы

Block анализируется через analyzeBlock(). Для блока создаётся новая область видимости table.enterScope(),
после чего каждый stmt внутри блока передаётся обратно в analyzeStmt(), а в конце область закрывается
через table.exitScope(). Поэтому локальные переменные блока не попадают во внешний scope, но все ошибки
внутренних инструкций собираются тем же общим механизмом.

If сначала анализирует condition через analyzeExpr(). Условие обязано иметь тип bool, иначе выдаётся
ошибка if condition must be bool с фактическим типом. После проверки условия thenBranch анализируется
как обычный statement через analyzeStmt(), а если есть elseBranch, он анализируется тем же способом.
Отдельного scope сам if не создаёт: scope появляется только если ветка является Block.

While работает почти так же с условием: condition проходит analyzeExpr(), и результат должен быть bool,
иначе выдаётся while condition must be bool. Перед анализом тела loopDepth увеличивается на 1, затем
body передаётся в analyzeStmt(), а после выхода loopDepth уменьшается обратно. Этот счётчик нужен,
чтобы break и continue понимали, находятся ли они внутри цикла.

Break не анализирует выражений и не меняет таблицу символов. Единственная проверка — loopDepth: если он
равен 0, значит break стоит вне цикла, и анализатор выдаёт ошибку 'break' outside of loop. Внутри while
значение loopDepth больше нуля, поэтому break считается допустимым.

Continue проверяется тем же правилом, что и break. Если loopDepth равен 0, continue находится вне цикла,
и анализатор выдаёт 'continue' outside of loop. Если текущая глубина цикла больше нуля, инструкция
считается корректной и дополнительных действий не требует.

ExprStmt — это выражение, используемое как самостоятельная инструкция, например вызов функции print(x).
Семантический анализатор просто вызывает analyzeExpr(exprStmt->expr), чтобы проверить само выражение,
разрешить его символы, типы, аргументы вызовов и возможные побочные ошибки. Возвращаемый тип выражения
здесь отдельно не используется.

Return сверяется с currentReturnType, который выставляется при анализе тела функции, метода,
конструктора или деструктора. Если return содержит value, а текущий тип возврата void, выражение всё
равно анализируется для сбора вложенных ошибок, после чего выдаётся void function cannot return a value.
Если функция не void, value анализируется с currentReturnType как ожидаемым типом, и результат должен
неявно приводиться к типу возврата, иначе появляется return type mismatch. Если return не содержит
значения, но currentReturnType не void, анализатор выдаёт return without value in non-void function.

##  Функции

FuncDecl на четвёртом проходе уже имеет имя и сигнатуру в таблице символов после collectTopLevel(), поэтому
analyzeStmt() здесь занимается только анализом тела функции. Сначала анализатор ищет символ функции через
table.resolve(func->name), сохраняет предыдущий currentReturnType и выставляет текущий тип возврата из
sym->funcInfo->returnType. Это нужно для всех return внутри тела: они будут сравниваться именно с типом
возврата текущей функции. Если символ или funcInfo по какой-то причине не найдены, currentReturnType
ставится в nullptr, чтобы не делать ложных проверок возврата.

Тело функции анализируется в отдельном scope. Анализатор вызывает table.enterScope(), затем по сигнатуре
из funcInfo создаёт символы параметров как обычные Variable. Каждый параметр считается уже
инициализированным, получает свой const-флаг и тип из ранее собранного списка params. После успешного
table.declare() AST параметр получает resolvedSym. Затем анализатор проходит по func->body->statements и
передаёт каждый statement в analyzeStmt(). Сам Block функции здесь не вызывает дополнительный enterScope,
потому что область функции уже открыта вручную для параметров и тела.

После анализа всех инструкций table.exitScope() закрывает область функции. Если currentReturnType
существует и не является void, анализатор дополнительно вызывает alwaysReturns(func->body). Эта проверка
нужна, чтобы non-void функция возвращала значение или гарантированно завершала выполнение на всех путях.
Завершающими инструкциями считаются return, а также expression statement с вызовом exit(...) или
panic(...), потому что эти builtin-функции не возвращают управление. Если гарантированного завершения
нет, выдаётся ошибка function does not return a value on all paths. В конце currentReturnType
возвращается к предыдущему значению, чтобы вложенный анализ не испортил контекст внешней функции или
метода.

StructDecl и TypeAlias сразу после функции на этом проходе специально пропускаются. Их имена были
зарегистрированы в predeclareTopLevel(), а полная информация о полях структуры и aliasTarget была
собрана в collectTopLevel(). У структуры и синонима типа нет тела, которое нужно исполнять или
дополнительно обходить как statement, поэтому повторный анализ здесь только продублировал бы уже
выполненную работу.

##  Классы

ClassDecls к этому моменту уже почти полностью собраны в collectTopLevel(): поля, методы, вложенные
структуры, конструктор и деструктор уже лежат в classInfo. Поэтому analyzeStmt() сначала находит символ
класса через table.resolve(clas->name), проверяет что у него есть classInfo, и если данных нет — просто
выходит из ветки. Затем создаётся тип self как TypeKind::Class с именем текущего класса. Поля класса
не объявляются как локальные переменные в scope метода: вместо этого анализатор сохраняет предыдущий
currentClass и ставит currentClass = classSym->classInfo, чтобы обращения к полям могли разрешаться как
неявный self.field.

Для каждого метода класса анализатор берёт уже собранную сигнатуру из classInfo->methods и временно
выставляет currentReturnType в тип возврата метода. Затем открывается новый scope метода, в него
добавляется единственное явно созданное имя self, помеченное как инициализированная переменная типа
текущего класса. После self в scope добавляются параметры метода: каждый параметр получает kind =
Variable, isInitialized = true, свой const-флаг и тип из FuncInfo, а AST параметра получает resolvedSym.
Дальше все инструкции method->body->statements последовательно передаются в analyzeStmt(). После выхода
из scope, если метод не void, alwaysReturns(method->body) проверяет что значение возвращается или
выполнение гарантированно завершается через exit(...) или panic(...) на всех путях; иначе выдаётся ошибка
method does not return a value on all paths. В конце для метода восстанавливается предыдущий
currentReturnType.

Конструктор анализируется похожим образом, но его currentReturnType всегда ставится в void. Для него
также открывается отдельный scope, объявляется self, затем регистрируются параметры конструктора из
classInfo->constructor и анализируются все инструкции тела. После этого scope закрывается, а прежний
currentReturnType восстанавливается. Деструктор ещё проще: он тоже анализируется с return type void,
получает scope с self, после чего анализируются инструкции destructor->body. Параметры деструктора на
этом этапе уже не добавляются, потому что их отсутствие проверялось раньше при сборе класса.

После методов, конструктора и деструктора currentClass возвращается к предыдущему значению. Это важно
для вложенного анализа: пока currentClass установлен, Identifier может резолвиться как поле текущего
класса, а вызов метода может находиться через classInfo. После выхода из ClassDecl такой контекст больше
не должен влиять на обычные функции или следующий top-level код.

NamespaceDecl на четвёртом проходе использует scope, который был создан и заполнен на предыдущих
проходах. Анализатор ищет символ namespace, проверяет namespaceScope, входит в него через pushScope(),
сохраняет предыдущий currentNamespace и дописывает имя текущего namespace. После этого все вложенные
decls анализируются обычным analyzeStmt(). В конце currentNamespace возвращается назад, а scope
закрывается через exitScope().

ExportDecl на этом этапе работает как обёртка над обычной декларацией: analyzeStmt() сначала запускается
для exp->decl, чтобы тело или инициализатор экспортируемой сущности прошли те же проверки, что и без
export. Дополнительная работа нужна только для VarDecl: глобальные переменные окончательно
регистрируются и связываются с символами именно в этой ветке четвёртого прохода, поэтому после анализа
export var анализатор отдельно проходит по var->vars, находит каждый символ и выставляет isExported =
true. Для функций, структур, классов, alias и namespace флаг экспорта уже был проставлен в
collectTopLevel().

##  Разбор выражений - analyzeExpr

Новый этап семантического анализа начинается с analyzeExpr(). Эта функция получает само выражение и
опциональный expected тип, который передаётся сверху из контекста: инициализации переменной, return,
присваивания, аргумента функции или другого выражения. Если выражение отсутствует, функция сразу возвращает
nullptr. Такой случай допустим, например, для объявления без инициализатора, где анализировать пока
нечего.

Числовой литерал Number сначала делится на float и integer по флагу isFloat. Для float-литерала, если
снаружи передан expected и он является float-типом, resolvedType берётся прямо из expected. Это позволяет
float32 x = 1.5 сохранить литерал как float32. Если подходящего контекста нет, дробный литерал получает
тип по умолчанию Float64. Для целочисленного литерала контекст работает похожим образом: если expected
является int-типом, анализатор дополнительно вызывает integerLiteralFitsType() и проверяет, помещается ли
значение в этот конкретный тип. При переполнении выдаётся ошибка integer literal does not fit into type,
но resolvedType всё равно ставится в expected. Если expected является float-типом, целое число также
принимает этот тип, то есть случай float x = 5 обрабатывается как 5.0. Если контекста нет, integer
получает тип по умолчанию Int32.

Строковый литерал String обычно получает тип String, но у него есть короткая контекстная проверка: если
expected равен Char и длина строки равна 1, литерал считается Char. Это позволяет одиночную строку
использовать там, где ожидается символ. Во всех остальных случаях строковый литерал остаётся String и
записывает этот тип в resolvedType.

Булевы и символьные литералы обрабатываются напрямую. Bool всегда получает TypeKind::Bool, а
CharLiteral всегда получает TypeKind::Char. Для них expected не меняет результат: если далее такой
литерал окажется несовместим с ожидаемым типом, ошибка будет выдана внешней проверкой приведения или
проверкой оператора.

NullLiteral представляет пустую ссылку на объект класса. Если analyzeExpr() вызван с expected и этот
тип является Class, null принимает именно этот class-тип и записывает его в resolvedType. Если class
контекста нет, null остаётся отдельным TypeKind::Null. Это согласуется с правилом приведения: null можно
использовать как пустую ссылку на class, но не как значение произвольного primitive или struct типа.

Identifier анализируется сложнее, потому что имя может обозначать не только переменную. Сначала
анализатор ищет символ через table.resolve(id->name). Если символ не найден, но сейчас анализируется
метод, конструктор или деструктор класса, проверяется currentClass: поля класса не объявляются как
локальные имена в scope, поэтому имя поля может быть разрешено как неявное self.field. В этом случае
id->resolvedField указывает на FieldInfo, а resolvedType становится типом поля. Если имя не найдено ни в
таблице, ни среди полей текущего класса, выдаётся ошибка is not declared.

Если символ найден, он должен быть именно Variable. Функция, структура, класс, type alias или namespace
не являются значениями выражения, поэтому для них вызывается nonValueSymbolMessage() и выдаётся
сообщение вида function is not a value, struct type is not a value или namespace is not a value. Для
переменной анализатор вызывает ensureVariableTypeKnown(). Эта функция возвращает уже известный type, а
для auto переменных при необходимости лениво анализирует autoInit, защищается от циклического вывода
через isResolvingAuto и сохраняет выведенный тип обратно в символ. Если тип вывести нельзя, выражение
считается неразрешённым.

После ensureVariableTypeKnown() дополнительно проверяется, что sym->type действительно появился. Если
тип всё ещё неизвестен, анализатор выдаёт cannot use variable before its type is known. Иначе Identifier
получает id->resolvedSym = sym, а expr->resolvedType становится knownType. Использование
неинициализированной переменной здесь не считается ошибкой по спецификации: такая переменная
автоматически получает default-значение своего типа, поэтому analyzeExpr() для Identifier проверяет
именно существование имени и тип, а не флаг isInitialized.

##  Бинарные операции

Binary начинается с рекурсивного анализа обоих операндов. Контекст expected пробрасывается вниз,
чтобы литералы могли учитывать ожидаемый внешний тип. Если хотя бы один
операнд не получил тип, анализатор сразу возвращает nullptr, потому что дальнейшая проверка
совместимости уже не имеет смысла.

Оператор Add сначала проверяет специальные случаи строк. Если оба операнда имеют тип String, результатом
становится String. Также разрешены string + char и char + string: символ склеивается со строкой как
однобайтовая строка, а resolvedType всего выражения снова становится String. Если строковый случай не
подошёл, Add проваливается в общую арифметическую ветку вместе с Sub, Mul, Div, Mod и Pow.

Арифметические операции требуют, чтобы оба операнда были numeric. Если хотя бы один тип не числовой,
выдаётся ошибка arithmetic operator requires numeric operands. Для Mod есть дополнительное ограничение:
оба операнда должны быть именно целочисленными, иначе оператор '%' выдаёт ошибку requires integer
operands. После этого анализатор вызывает commonType(leftType, rightType), который выбирает общий тип
для операции: одинаковые типы остаются без изменений, int + float поднимается к float, два signed или
два unsigned int выбирают больший тип, а signed + unsigned подбирает signed-тип достаточной ширины или
Float64, если целого общего типа не хватает. Если общий тип получить нельзя, выдаётся incompatible types
in arithmetic.

Для Pow есть отдельная корректировка результата. После вычисления commonType() анализатор пытается
вычислить правый операнд как константное целое выражение через evalConstIntExpr(). Если степень
оказалась отрицательной, а общий тип был integer, результат операции принудительно становится Float64,
потому что отрицательная степень даёт дробное значение. В остальных случаях resolvedType арифметической
операции становится найденным common type.

Операторы Less, Greater, LessEqual и GreaterEqual работают только с numeric операндами. Сначала
проверяется, что оба типа числовые, затем дополнительно проверяется наличие общего типа через
commonType(). Если типы несовместимы, выдаётся ошибка incompatible types in comparison. При успешной
проверке результат любого такого сравнения всегда Bool.

Операторы EqualEqual и NotEqual имеют более широкие правила. Сначала isUnsupportedEqualityType()
запрещает сравнение struct, array и dynamic array: для них оператор == или != не определён. Если оба
операнда numeric, они сравниваются через commonType(), как и в обычных числовых сравнениях. Для
остальных типов анализатор требует, чтобы хотя бы один тип неявно приводился к другому через
isImplicitlyConvertible(). Если такого привидения нет, выдаётся cannot compare left with right. При 
успешной проверке результат равенства всегда Bool.

Логические операторы And и Or принимают только Bool с обеих сторон. Если leftType или rightType не Bool,
анализатор выдаёт logical operator requires bool operands. При корректных операндах resolvedType
становится Bool. Если оператор не попал ни в одну известную ветку switch, analyzeExpr() возвращает
nullptr.

##  Унарные операции

Unary начинается с подготовки ожидаемого типа для операнда. Если оператор — UnaryMinus или UnaryPlus, а
снаружи передан numeric expected, анализатор пробрасывает этот expected внутрь operandExpected. Это
нужно, чтобы выражения вроде int8 x = -5 проверялись как отрицательный литерал именно в контексте int8,
а не как отдельный Int32 до применения минуса. После этого операнд анализируется через
analyzeExpr(unary->operand, operandExpected).
 
UnaryMinus требует numeric операнд. Если операнд не числовой, выдаётся unary - requires numeric operand.
Для отрицания целочисленного литерала есть отдельная проверка переполнения: если снаружи был expected,
операнд является Number и он не float, анализатор вручную считает negValue = -num->value и проверяет его
через integerLiteralFitsType(negValue, expected). Если отрицательное значение не помещается в ожидаемый
тип, выдаётся integer literal does not fit in type. При успешной проверке resolvedType становится
expected. Во всех остальных случаях результат унарного минуса получает тип самого операнда.

UnaryPlus также принимает только numeric операнд. Он не меняет значение и не делает дополнительных
преобразований: после проверки типа resolvedType просто становится operandType. Если операнд не
числовой, анализатор выдаёт unary + requires numeric operand.

Оператор Not работает только с bool. Операнд анализируется обычным образом, после чего его kind должен
быть TypeKind::Bool. Если это не так, выдаётся ошибка '! requires bool operand'. При корректном операнде
результат логического отрицания всегда имеет тип Bool.

Increment и Decrement используются для ++ и --. Сначала проверяется, что operandType является
целочисленным: для float, string, bool и других типов выдаётся ++/-- requires integer operand. Затем
операнд должен быть lvalue, потому что изменить можно только реальное место хранения: переменную, поле,
элемент массива или namespace-переменную. Если операндом является вызов функции, литерал или результат
выражения вроде a + b, isLvalue() возвращает false и анализатор выдаёт ++/-- requires an lvalue operand.

После lvalue выполняются проверки const. resolveTargetRoot() спускается к корневому символу операнда, и
если этот символ const, ++/-- запрещается ошибкой cannot apply ++/-- to const. Отдельно проверяется
Identifier, который мог быть неявным self-полем класса: если id->resolvedField существует и само поле
const, выдаётся cannot apply ++/-- to const field. При успешных проверках resolvedType операции
становится operandType.

##  Вызов метода класса

Вызов метода класса попадает в ветку FuncCall, когда callee является FieldAccess, то есть выражение
имеет вид obj.method(args...). Сначала анализатор разбирает object у FieldAccess через analyzeExpr().
Если тип объекта не найден или найденный тип не является Class, вызов метода невозможен. При этом
аргументы всё равно отдельно прогоняются через analyzeExpr(), чтобы собрать ошибки внутри них, и только
после этого для известного неклассового типа выдаётся method call on non-class type.

Если object действительно имеет class-тип, анализатор ищет символ класса через resolveQualifiedSymbol()
по имени objType->name. Этот helper умеет работать как с обычным именем, так и с квалифицированным
именем через "::": если разделителя нет, он ищет символ в текущей таблице, а внутри currentClass также
может вернуть вложенную структуру; если "::" есть, путь делится на namespace/class часть и последний
member, после чего поиск продолжается уже как квалифицированный. Для вызова метода здесь важно получить
именно classSym с заполненным classInfo, потому что список методов лежит в classInfo->methods.

Имя метода берётся из fieldCallee->field. Анализатор ищет его в classInfo->methods. Если такого метода
нет, выдаётся ошибка class has no method, но аргументы снова анализируются отдельно для сбора их
собственных ошибок. Если метод найден, его FuncInfo сохраняется сразу в двух местах: call->resolvedMethod
и fieldCallee->resolvedMethod. Это связывает AST вызова и AST доступа к полю с найденной сигнатурой
метода, чтобы следующие этапы не искали метод повторно.

Перед проверкой аргументов вызывается appendMissingDefaultArgs(). Этот helper не работает для variadic
методов и ничего не делает, если аргументов уже не меньше, чем параметров. Иначе он проходит по
недостающим параметрам и добавляет в call->args либо defaultValue параметра, либо сгенерированное
makeDefaultExprForType() значение по типу параметра. Если default-аргумент создать нельзя, добавляется
ошибка cannot create default argument for parameter. После этого вызов дальше проверяется уже так, будто
недостающие аргументы были явно переданы.

Дальше анализатор строит список argTypes. Для каждого аргумента выбирается expectedArg: если для этой
позиции существует параметр метода, expectedArg берётся из methodInfo->params[i].type, иначе остаётся
nullptr. Затем аргумент анализируется через analyzeExpr(call->args[i], expectedArg), чтобы литералы и
сложные выражения могли типизироваться в контексте ожидаемого параметра.

Финальная проверка делается через checkCallArguments(). Для метода туда передаётся строка вида
method 'name', список ParamInfo, собранные argTypes, флаг isVariadic и позиция вызова. Helper сначала
проверяет количество аргументов: для обычного метода аргументов должно быть не больше числа параметров,
а для variadic — не меньше числа фиксированных параметров. Затем для всех предоставленных фиксированных
аргументов сравнивается фактический тип с ожидаемым типом параметра через isImplicitlyConvertible().
Если приведение невозможно, выдаётся ошибка method argument N: cannot convert actual to expected. После
успешной проверки resolvedType всего FuncCall становится returnType найденного метода.

##  Вызов функции

После ветки вызова метода через obj.method(...) анализатор переходит к прямому вызову по имени. Здесь
поддерживаются три формы: обычный f(...), вызов из namespace через Namespace::f(...), а также более
длинный квалифицированный путь A::B::f(...). Для Identifier calleeName становится именем идентификатора,
а символ ищется через table.resolve(). Для NamespaceAccess calleeName собирается как namespace::member,
а символ ищется через resolveQualifiedSymbol(), чтобы пройти по сохранённым scope пространств имён или
других квалифицированных сущностей.

Если callee не является ни Identifier, ни NamespaceAccess, такой объект нельзя вызвать как функцию.
Анализатор всё равно сначала вызывает analyzeExpr(call->callee), затем отдельно анализирует все
аргументы, чтобы собрать возможные ошибки внутри них. После этого выдаётся expression is not callable и
разбор вызова заканчивается.

Если прямой поиск символа не дал результата, но сейчас анализируется тело метода, конструктора или
деструктора класса, включается специальная проверка неявного метода текущего класса. Анализатор смотрит
currentClass->methods и пытается найти метод с именем calleeName. Если метод найден, он обрабатывается
почти так же, как obj.method(...), только без явного объекта self: FuncInfo записывается в
call->resolvedMethod, затем appendMissingDefaultArgs() добавляет недостающие default-аргументы, каждый
аргумент анализируется с expected типом соответствующего параметра, а checkCallArguments() проверяет
количество и приводимость типов. Результатом всего вызова становится methodInfo->returnType.

Если символ всё ещё не найден, анализатор выдаёт ошибку name is not declared. Если символ найден, но
это Class, выражение трактуется как создание экземпляра класса через ClassName(args...). В call->resolvedCallee
сохраняется символ класса, затем берётся classInfo->constructor, если он есть. При наличии конструктора
сначала добавляются default-аргументы через appendMissingDefaultArgs(), затем все аргументы анализируются
с expected типами параметров конструктора и проверяются через checkCallArguments() с сообщением
constructor of ClassName. Если конструктора нет, допускается только вызов без аргументов: наличие
любых argTypes приводит к ошибке class has no constructor; default constructor takes no arguments.
Результирующим типом выражения становится sym->type, то есть тип создаваемого класса.

Если найденный символ не Class и не Function, вызов невозможен, поэтому выдаётся name is not a
function. Для настоящей функции call->resolvedCallee получает найденный символ. Далее для обычных
пользовательских функций, у которых есть funcInfo, анализатор добавляет недостающие default-аргументы
по сигнатуре. Builtin-функции print, len, push, pop, input, exit и panic из этой общей default-логики
исключаются, потому что ниже в analyzeExpr() для них есть отдельные специализированные ветки с
собственными правилами проверки аргументов и типов:

1) push(elem, arr) проверяется отдельно, потому что тип первого аргумента зависит от типа второго. 

Сначала проверяется, что аргументов ровно два. Затем вторым аргументом анализируется массив, и он обязан 
быть изменяемым lvalue: нельзя передать временное выражение или const-массив. После этого тип второго 
аргумента должен быть DynArray. Если это не динамический массив, первый аргумент анализируется без 
expected-типа и вызов завершается с void. Если массив корректный, первый аргумент анализируется с 
expected = elementType массива, а затем проверяется isImplicitlyConvertible() от типа элемента к типу 
элемента массива. При несовместимости выдаётся push element type mismatch. Успешный push всегда имеет 
тип void.

После push для остальных вызовов анализатор собирает общий список argTypes. Каждый аргумент проходит
analyzeExpr(), а expectedArg берётся из funcInfo только для обычных функций, кроме print, len и pop,
потому что эти builtin-функции проверяют аргументы по своим правилам. Дальше идут отдельные проверки
по имени calleeName.

2) input() не принимает аргументов и имеет контекстный возвращаемый тип. 

Если expected не передан сверху, анализатор не может понять, что именно
читать, и выдаёт input requires an expected type. Даже при наличии expected тип должен быть поддержан
вводом: string, char, целые, float, а также массивы или динамические массивы таких элементов. Если тип
не поддержан, выдаётся input cannot read value of type. При успехе resolvedType становится expected.

3) print(x1, x2, ...) требует хотя бы один аргумент.

Каждый аргумент уже проанализирован, и дальше проверяется только его kind. Печатать разрешено bool,
string, char, все int/uint, float32/float64, Array и DynArray. Для остальных типов выдаётся print 
argument N has unsupported type. Возвращаемый тип print всегда void.

4) len(x) требует ровно один аргумент. 

Если аргумент есть, его тип должен быть Array, DynArray или String; для других типов
выдаётся len argument must be an array or string. Независимо от найденных ошибок resolvedType у len
ставится Int32.

5) pop(arr) требует ровно один аргумент.

Аргумент обязан быть изменяемым lvalue, потому что pop меняет
массив, и не может ссылаться на const-корень. Затем тип аргумента должен быть DynArray, иначе выдаётся
pop argument must be a dynamic array. Если тип динамического массива известен, результатом pop становится
elementType этого массива; если тип не удалось получить, resolvedType остаётся nullptr.

6) exit(code) требует ровно один аргумент.

Аргумент анализируется с ожидаемым типом int32. Если количество аргументов не равно одному, выдаётся
exit expects 1 argument. Если тип аргумента известен, но не приводится к int32 через
isImplicitlyConvertible(), выдаётся ошибка exit argument 1: cannot convert. Возвращаемый тип exit всегда
void, но для alwaysReturns() вызов exit(...) считается завершающей инструкцией.

7) panic(msg) требует ровно один аргумент.

Аргумент анализируется с ожидаемым типом string. Если количество аргументов не равно одному, выдаётся
panic expects 1 argument. Если тип аргумента известен, но не приводится к string, выдаётся ошибка panic
argument 1: cannot convert. Возвращаемый тип panic всегда void, но для alwaysReturns() вызов panic(...)
считается завершающей инструкцией.

После всех специальных builtin-веток обычные функции проверяются общим путём. Если у символа есть
funcInfo, вызывается checkCallArguments("function 'name'", ...). Для extern-C функций этот helper
включает режим маршалинга: string, array, dynarray и pointer-подобные значения могут быть представлены
как C-совместимые значения, а variadic-хвост дополнительно проверяется на ABI-представимость. Затем
анализатор пробрасывает в AST флаги call->isExternC и call->isVariadic, чтобы кодоген видел, как именно
вызывать функцию. resolvedType берётся из funcInfo->returnType, а если funcInfo нет, используется
sym->type. После этого обработка FuncCall завершается возвратом expr->resolvedType.

##  Доступ к полю структуры или класса

FieldAccess обрабатывает выражения вида object.field. Сначала анализатор проверяет специальные случаи,
где слева стоит не объект, а сам тип. Если object является Identifier и это имя разрешается в SymbolKind
Struct или Class, выражение трактуется как чтение default-значения поля типа: TypeName.field. Для поиска
поля используется helper findFieldInTypeSymbol(), который берёт список fields из structInfo или classInfo
символа и возвращает FieldInfo с нужным именем. Если поля нет, выдаётся type has no default field. Если
поле найдено, FieldAccess получает resolvedField, флаг isTypeDefaultFieldAccess = true, а resolvedType
становится типом найденного поля.

Такой же специальный случай поддержан для квалифицированного типа из namespace: Namespace::TypeName.field.
Если object является NamespaceAccess, анализатор ищет тип через resolveQualifiedSymbol(ns->nameSpace,
ns->member), собирает полное имя для сообщения об ошибке и снова проверяет, что найденный символ является
Struct или Class. Дальше логика та же: findFieldInTypeSymbol(), ошибка при отсутствии default-поля,
заполнение resolvedField, isTypeDefaultFieldAccess и resolvedType.

Если слева не найден тип, начинается обычный доступ к полю объекта. Анализатор вызывает
analyzeExpr(field->object), получает objType и дальше работает только с Struct или Class. Для Struct
символ структуры ищется через resolveQualifiedSymbol(objType->name), затем перебираются
structInfo->fields. При совпадении имени поле записывается в field->resolvedField, а resolvedType
выражения становится типом поля. Если структура найдена, но нужного поля нет, выдаётся struct has no
field.

Для Class логика похожая: classSym ищется через resolveQualifiedSymbol(objType->name), затем
проверяются classInfo->fields. Если поле найдено, оно сохраняется в resolvedField, а тип выражения
становится типом поля. Если поля нет, анализатор дополнительно смотрит classInfo->methods. Метод нельзя
читать как значение через obj.method без скобок, поэтому при найденном методе выдаётся method is not a
value; call it with (). Если нет ни поля, ни метода, выдаётся class has no field or method.

Если objType существует, но не является ни Struct, ни Class, доступ к полю запрещается ошибкой field
access on non-struct/class type. Таким образом FieldAccess отделяет три разных случая: чтение
default-поля у типа, чтение поля у значения struct/class и ошибочную попытку обратиться к полю у
primitive, массива, строки или другого неподходящего выражения.

##  Индексация и литералы массива

ArrayAccess обрабатывает выражение arr[i]. Сначала анализатор отдельно разбирает объект индексации и 
индекс через analyzeExpr(). Объектом может быть Array, DynArray или
String. Если тип объекта не входит в этот список, выдаётся index operator on non-array type. Индекс при
этом должен быть целочисленным: если indexType известен и не проходит isIntType(), анализатор выдаёт
array index must be integer. Результат индексации зависит от объекта: для String элементом считается
Char, а для Array и DynArray результатом становится elementType массива.

ArrayLiteral обрабатывает выражения вида [expr1, expr2, ...]. В начале анализатор смотрит expected:
если снаружи ожидается Array или DynArray, то expected->elementType становится targetElemType, то есть
эталонным типом для всех элементов литерала. Это позволяет, например, в контексте float[] проверить
[3.14, 5, 6.2] так, чтобы целое 5 анализировалось как значение, приводимое к float.

Пустой массив можно типизировать только из контекста. Если elements пустой, но expected не является
массивом или у него нет elementType, анализатор выдаёт cannot infer type of empty array literal. Если
пустой литерал стоит в контексте DynArray, результатом сразу становится makeDynArrayType(targetElemType),
то есть пустой динамический массив нужного типа.

Если массив непустой и targetElemType ещё не задан контекстом, анализатор выводит его из элементов.
Первый элемент анализируется без expected и становится начальным targetElemType. Затем остальные
элементы также анализируются без expected, а их типы объединяются через commonType(). Если commonType()
не может подобрать общий тип, выдаётся array element N: incompatible types. Если общий тип найден, он
становится новым targetElemType, поэтому смешанные числовые литералы могут расширить тип всего массива.

После выбора targetElemType анализатор ещё раз проходит все элементы, теперь уже с expected =
targetElemType. Для каждого элемента проверяется isImplicitlyConvertible(elemType, targetElemType). Если
элемент нельзя привести к выбранному типу массива, выдаётся array element N: cannot convert. При этом
resolvedType каждого элемента принудительно выставляется в targetElemType, чтобы дальнейшие этапы видели
элементы уже в согласованном типе массива.

Финальный тип литерала зависит от expected. Если expected был DynArray, литерал получает
makeDynArrayType(targetElemType). Если expected был фиксированным Array, анализатор дополнительно
сравнивает количество элементов литерала с expected->arraySize. При несовпадении выдаётся array literal
size mismatch, а при успехе resolvedType становится ровно expected. Если expected не было, литерал
считается динамическим массивом makeDynArrayType(targetElemType). Поэтому auto arr = [1, 2, 3] выводится
как динамический массив int, а вложенные литералы по той же логике дают многомерные динамические массивы.

##  Литерал структуры, cast и доступ через namespace

StructLiteral обрабатывает выражения вида Point { x: 5, y: 10 }. Сначала имя структуры разрешается через
resolveQualifiedSymbol(structLit->name). Если символ не найден или найденная сущность не является
Struct, анализатор выдаёт is not a struct type. Если structInfo существует, дальше проверяются явно
указанные поля литерала. Для этого используется множество seen: если одно и то же поле указано два раза,
выдаётся field is already initialized, но значение всё равно анализируется через analyzeExpr(), чтобы
собрать возможные ошибки внутри выражения.

Для каждого поля литерала анализатор ищет одноимённый FieldInfo в sym->structInfo->fields. Если поля нет
в структуре, выдаётся struct has no field, а значение снова анализируется без expected-типа только для
сбора ошибок. Если поле найдено, его тип передаётся в analyzeExpr(init.value, foundField->type) как
expected. После анализа проверяется isImplicitlyConvertible() от типа значения к типу поля; при
несовместимости выдаётся field cannot convert. Отсутствующие поля здесь специально не достраиваются:
codegen сначала копирует runtime default-слоты структуры, а затем накладывает явно указанные поля, поэтому
переопределение Type.field влияет на последующие runtime-инициализации. Результатом StructLiteral
становится TypeKind::Struct с именем найденной структуры.

CastExpr отвечает за явное приведение cast<T>(expr). Сначала анализируется исходное выражение
cast->value, затем целевой тип разрешается через resolveTypeName(cast->targetType). Если целевой тип
неизвестен, выдаётся unknown type in cast. Если исходный тип получен, но isCastable(fromType, targetType)
возвращает false, анализатор выдаёт cannot cast from to target. При этом resolvedType самого cast всё
равно становится targetType, чтобы дальнейший анализ видел заявленный результат явного приведения.

NamespaceAccess обрабатывает доступ к значению через namespace::member. Символ ищется через
resolveQualifiedSymbol(namespase->nameSpace, namespase->member), а для сообщений собирается fullName
вида namespace::member. Если символ не найден, выдаётся member is not declared in namespace. Если символ
найден, но это не Variable, анализатор вызывает nonValueSymbolMessage(): функцию, структуру, класс,
alias или сам namespace нельзя использовать как значение без соответствующего синтаксиса. Для переменной
тип дополнительно доводится через ensureVariableTypeKnown(), что важно для auto-переменных. После
успешного разрешения NamespaceAccess получает resolvedSym, а resolvedType выражения становится известным
типом найденной переменной. Если ни одна ветка analyzeExpr() не подошла, функция возвращает nullptr.

##  Краткое описание всех helper функций

Ниже перечислены helper-функции, которые используются во многих ветках анализатора, но не являются
отдельными проходами или самостоятельными узлами AST. Они держат общие правила типов, имён, вызовов и
импортов в одном месте, чтобы analyzeStmt() и analyzeExpr() не дублировали одну и ту же логику.

* makeType(), makeArrayType(), makeDynArrayType() 

Используются для создания объектов Type для примитивов, фиксированных
массивов и динамических массивов. Для массивов сразу заполняются elementType и arraySize: у 
фиксированного массива размер хранится явно, у динамического ставится -1.

* typesEqual() 

Сравнение двух Type по смыслу, а не по адресу shared_ptr. Для примитивов достаточно
одинакового kind, для Array дополнительно сравниваются elementType и arraySize, для DynArray —
elementType, а для Struct, Class и Alias сравнивается name. 

* typeToString(), typeNameToString() 

Превращают уже разрешённый Type или синтаксический TypeName в строку для сообщений об
ошибках.

* isIntType(), isFloatType(), isNumericType(), isSignedIntType() и isUnsignedIntType() 

Короткие предикаты над TypeKind. Они используются в арифметике, cast, input, print, проверках индексов,
инкрементах и приведениях. isInputSupportedType() отдельно описывает типы, которые умеет читать input():
строки, char, числовые типы, а также массивы и динамические массивы из таких элементов.

* typeRank(), intBitWidth() и signedIntKindForBits() 

Используются для обслуживания числовых расширений. typeRank() задаёт ранг типов и
порядок widening-приведений внутри signed, unsigned и float-групп. intBitWidth() возвращает битность
целого типа, а signedIntKindForBits() подбирает signed-тип нужной ширины для смешанных signed/unsigned
операций.

* isImplicitlyConvertible()

Используется для проверки неявных приведений. Разрешены одинаковые типы, widening внутри
signed-целых, widening внутри unsigned-целых, float32 в float64, любые целые в float, фиксированный Array
в DynArray с рекурсивной проверкой элемента и null в Class. Всё, что не входит в этот список, требует
явного cast или считается ошибкой типов.

* isCastable() 

Описывает более широкие правила явного cast<T>(expr). Он разрешает приведения между
числовыми типами, int и bool, char и int, а также string и bool в обе стороны. При этом string в число и
сложные struct/class/array преобразования не считаются допустимыми.

* commonType() 

Выбирает общий тип для бинарных числовых операций и некоторых сравнений. Если типы
одинаковые, возвращается исходный тип. Для int/float выбирается float32 или float64, для двух float —
больший float, для двух signed или двух unsigned — больший целый. Для signed + unsigned подбирается
signed-тип, который может представить оба диапазона; если такого типа до int64 нет, результатом
становится float64. Если общий тип выбрать нельзя, возвращается nullptr.

* isUnsupportedEqualityType() 

запрещает прямое == и != для Struct, Array и DynArray. 

* integerLiteralFitsType()

Проверяет, помещается ли целочисленный литерал в заданный int, uint или char тип. Эта проверка нужна
особенно для маленьких типов вроде int8 и uint8, где default int32 литерал может не подходить по
диапазону.

* makeDefaultExprForType() 

Создаёт default-значения для переменной или default-аргумента.
Для bool создаётся false, для char — '0', для string — "NULL", для чисел — 0, для DynArray — пустой
ArrayLiteral, для фиксированного Array — ArrayLiteral из default-значений элемента, для Struct —
StructLiteral с нужным именем, а для Class — null. Для void default-значение не создаётся.

* makeParamInfo() 

Переносит синтаксический Param в ParamInfo: имя, тип, defaultValue и const-флаг. Этот
формат хранится в FuncInfo, MethodInfo и constructor-информации, чтобы дальнейшие проверки вызовов
работали не с AST параметров напрямую, а с уже разрешённой сигнатурой.

* findFieldInTypeSymbol() 

Ищет поле по имени внутри SymbolKind::Struct или SymbolKind::Class. Он выбирает
fields из structInfo или classInfo и возвращает указатель на FieldInfo. Этот helper используется и для
чтения default-поля через Type.field, и для переопределения default-значения поля.

* ensureVariableTypeKnown() 

Доводит тип auto-переменной до известного состояния. Если sym->type уже есть,
он возвращается сразу. Если это auto-переменная без типа, анализатор берёт сохранённый autoInit,
анализирует его выражение и записывает выведенный тип обратно в Symbol. Флаг isResolvingAuto защищает от
циклов вида auto a = a. Для const integer auto дополнительно вычисляется intConstValue.

* ensureAliasTypeKnown() 

Лениво разрешает type alias. Если alias уже имеет type, он возвращается сразу.
Иначе helper берёт aliasTarget, вызывает resolveTypeName() и сохраняет результат. Флаг isResolvingAlias
ловит циклические alias-цепочки, например когда A ссылается на B, а B обратно на A.

* resolveArrayTypeSuffix() 

Применяет один суффикс TypeName к уже известному elementType. Для [] создаётся
DynArray, для [expr] выражение размера вычисляется как константное целое через evalConstIntExpr(); размер
обязан быть положительным. resolveTypeName() строит полный Type из base-имени и всех suffixes: сначала
разрешается примитив, struct, class или alias, затем последовательно применяются массивные суффиксы.

* resolveDeclaredType() 

Используется в объявлениях переменных, параметров и полей. Он учитывает контекст:
auto разрешён не везде, const требует явного initializer в некоторых местах, void запрещён для
переменных и полей, а defaultValue при необходимости анализируется или создаётся по типу. Благодаря этому
collectTopLevel() и analyzeStmt() получают уже нормализованный тип объявления.

* evalConstIntExpr() 

Пытается вычислить выражение как целочисленную константу на этапе семантики. Он
поддерживает integer Number, const integer Identifier, const integer NamespaceAccess, унарные + и -, а
также бинарные +, -, *, /, %, Pow. 

* splitQualifiedName() 

Разбивает строку вида A::B::C на части, а appendQualifiedName() добавляет имя к
текущему namespace-префиксу. Две перегрузки resolveQualifiedSymbol() используют эти helpers для поиска
символа по обычному имени, по namespace + member или по уже собранной строке с "::". Поиск умеет идти
через namespaceScope и через вложенные структуры классов.

* isMarshalableToC() 

Проверяет, можно ли аргумент передать в extern-C функцию с ожидаемым типом. Для C
разрешается обычное isImplicitlyConvertible(), а также представление string, array, dynarray, class и
null как uint64-подобного значения. checkCallArguments() поверх этого проверяет количество аргументов,
variadic-хвост и приводимость каждого аргумента к параметру.

* appendMissingDefaultArgs() 

Достраивает недостающие аргументы вызова для обычных функций, методов и
конструкторов. Если у параметра есть defaultValue, используется оно; иначе helper пытается создать
default-выражение через makeDefaultExprForType(). Для variadic-вызовов helper ничего не добавляет.

* resolveTargetRoot() 

Спускается по lvalue-выражению к корневому Symbol. Для Identifier это сам символ, для
FieldAccess и ArrayAccess поиск продолжается через object, для NamespaceAccess используется
resolveQualifiedSymbol(). isLvalue() проверяет, можно ли выражение использовать слева от присваивания,
в ++/-- или как изменяемый массив в push/pop.

* nonValueSymbolMessage() 

Строит понятное сообщение для случаев, когда имя существует, но не является
значением выражения. Для Function он предлагает вызвать имя через (), для Struct и Class сообщает, что
это тип, для TypeAlias и Namespace тоже возвращает отдельные сообщения. error() добавляет сообщение в
общий список errors с текущим filePath, line и column; в конце analyzeModule() эти строки собираются в
один std::unexpected.

* alwaysReturns() 

Небольшой helper для non-void функций и методов проверяющий наличие return, exit или panic на всех путях 
завершения. Return считается завершением сразу. Block считается завершающим, если внутри есть 
завершающая инструкция. If считается завершающим только при наличии else и завершении обеих веток. 
Дополнительно expression statement с exit(...) или panic(...) также считается завершающим, потому что 
эти вызовы не возвращают управление.

* checkDuplicateParams(), checkDuplicateFields(), checkDuplicateMethods(), checkDuplicateNestedStructs()

Проверяют повторяющиеся имена внутри локального списка. Они не смотрят в таблицу символов, а работают
только с параметрами одной функции, полями одной структуры или класса, методами одного класса и
вложенными структурами одного класса. Для top-level имён используется declareTopLevelSymbol().


После завершения всех проходов analyzeModule() проверяет точку входа main, собирает накопленные ошибки и
возвращает std::unexpected, если семантика программы некорректна. Если ошибок нет, AST уже содержит
resolvedType, resolvedSym, resolvedField, resolvedCallee и другую привязочную информацию, поэтому main
передаёт тот же Program в CodeGen, где начинается генерация ассемблерного кода.

# Кодогенерация

Точка входа функция generate. Класс CodeGen на протяжении всей генерации держит открытыми 3 потока ввода:

1) text - код исполнения программы
2) rodata - строковые и константные литералы
3) bss - неинициализированные глобальные переменные

Эти потоки заполняются соответствующими инструкциями и в конце они склеиваются в один итоговый .asm 
файл.

Первым делом генерация объявляет через extern все встроенные в язык функции написанные напрямую на 
NASM в своём runtime и заполняет ими секцию text. Некоторые функции имеют префикс lang для того чтобы 
избежать конфликта с идентичными по имени функциями из libc. 

Затем в rodata сохраняются все возможные ошибки выполнения программы которые не удалось отследить на 
этапе компиляции, как например деление на ноль или выход за границы массива. Каждой ошибке соответствует 
своя метка для простоты вызова её из секции text. 

##  Регистрация смещений полей и классов

Первый полноценный проход кодогенерации собирает данные о layout
пользовательских типов. Происходит это в два обхода, сперва обход всех import
деклараций, а затем деклараций основного файла. Для каждого узла вызывается collectDecls(), которая
регистрирует структуры, классы, вложенные структуры классов, namespace, export и глобальные переменные.

Основная работа со смещениями находится в collectLayout(). На вход она получает полное имя структуры или
класса и вектор полей. Внутри создаются временные таблицы:

* layout - имя поля -> смещение от начала объекта
* types - имя поля -> уже разрешённый семантикой Type
* order - порядок объявления полей

Смещение начинается с 0. Для каждого поля берётся field.resolvedType, который должен быть уже выставлен
семантическим анализатором. Если типа нет, кодогенерация добавляет ошибку вида "field 'Type.field' has no
resolved type" и переходит к следующему полю. После этого через sizeOfType() вычисляется размер поля. Если
размер меньше или равен 0, это также считается ошибкой layout. (CodeGen.cpp :: 1522-1539)

Перед записью поля смещение выравнивается по размеру типа, но не больше 8 байт. Для этого считается остаток
offset % align, где align = min(size, 8). Если остаток не нулевой, к offset прибавляется недостающая часть
до следующей подходящей границы. После выравнивания текущее значение offset записывается как смещение поля,
тип поля сохраняется в types, а имя добавляется в order. Затем offset увеличивается на размер поля.

После обработки всех полей итоговый размер структуры или класса дополнительно выравнивается по 8 байтам.
Это нужно не только для одиночного объекта, но и для массивов структур, чтобы каждый следующий элемент
начинался с корректного адреса. Собранные данные переносятся в основные словари CodeGen:

* structLayouts[name] - имя типа -> {имя поля -> смещение}
* structFieldTypes[name] - имя типа -> {имя поля -> тип поля}
* structFieldOrder[name] - имя типа -> поля в порядке объявления
* structSizes[name] - имя типа -> итоговый размер объекта

Пример размещения структуры в памяти:

struct T {
   char a;  // 1 байт - достаточно выравнивания по 1
   int8 b;
   int c;   // 4 байта - достаточно выравнивания по 4
   float d; // 8 байт - выравнивание по 8
   int[3] arr;  // 3*4=12 байт, 12>8 -> 8 байт достаточно  
}

[ a ][ b ][___][___][ c ][ c ][ c ][ c ][ d ][ d ][ d ][ d ][ d ][ d ][ d ][ d ] ->
[arr][arr][arr][arr][arr][arr][arr][arr][arr][arr][arr][arr][___][___][___][___]

collectDecls() перед вызовом collectLayout() собирает полное имя через appendQualifiedName(), поэтому типы
внутри namespace регистрируются как Namespace::Type. Для StructDecl сразу регистрируется layout и сам
указатель декларации в structDecls, structDeclNames и structDeclsOrdered. Для ClassDecl выполняется такой
же сбор layout полей класса, после чего класс сохраняется в classDecls, classDeclNames и classDeclsOrdered.
Отдельно обходятся вложенные структуры класса: их имя строится как ClassName::NestedStruct, для них также
собираются смещения и сохраняется порядок объявления. 

ExportDecl не имеет собственного layout, поэтому collectDecls() рекурсивно запускается для вложенной
декларации. NamespaceDecl временно меняет currentNamespace, обходит все внутренние декларации и затем
возвращает прошлое значение namespace. Благодаря этому вложенные пространства имён корректно попадают в
полные имена типов и не конфликтуют с одноимёнными структурами или классами в другой области видимости.

После первого прохода generate проверяет список codegenErrors. Если ошибки layout уже накопились, генерация
останавливается и возвращает std::unexpected со всеми сообщениями. Если ошибок нет, эти таблицы дальше
используются почти во всех местах работы с пользовательскими типами: при доступе obj.field, при копировании
struct/class значений, при default-инициализации, при выделении .bss слотов для default-полей структур и
дефолтных экземпляров классов. Поэтому регистрация смещений должна идти до генерации .bss и до компиляции
основных инструкций программы. 

##  Резервирование слотов в .bss

После успешной регистрации layout CodeGen знает размеры структур, классов и полей, поэтому может заранее 
выделить память в секции bss. Эта память нужна для runtime default-значений и глобальных переменных.

Первый цикл проходит по structDeclsOrdered. Для каждого поля каждой структуры берётся полное имя структуры
из structDeclNames, а размер поля считается через sizeOfType(field.resolvedType). В bss создаётся метка:

`__default_<StructName>_<fieldName>: resb <size>`

Такие слоты создаются даже для полей без явного default-значения. Это нужно потому что синтаксис
StructName.field работает как обращение к runtime default-значению поля, а выражение StructName.field = x
должно иметь реальный адрес, куда можно записать новое значение. Позже, уже в начале main, эти слоты будут
заполнены либо пользовательскими default-выражениями, либо default-значениями самих типов. Вследствии чего
в генерации не используется секция data для хранения полей.

Для классов выделяется не отдельный слот на каждое поле, а один общий default-экземпляр класса:

`__default_instance_<ClassName>: resb <classSize>`

Размер берётся из structSizes, потому что layout классов хранится в тех же таблицах, что и layout структур.
Такой default-экземпляр хранит текущее значение всех полей класса и используется для обращений вида
MyClass.field и MyClass.method(). Когда позже создаётся новый объект класса, compileClassConstruction()
сначала копирует байты из этого default-экземпляра, а затем вызывает конструктор если он есть.

После структур и классов выделяются bss-слоты под глобальные переменные. Их список был собран раньше в
collectDecls() в globalVars и globalsBySymbol. Для каждой переменной по resolvedSym находится тип, затем
через sizeOfType() считается размер. Минимальный размер слота - 8 байт, чтобы скалярные значения,
указатели и маленькие типы имели единый безопасный размер записи. DynArray занимает 24 байта, так как
хранит ptr, len и cap.

На этом этапе в bss появляются только адреса и размеры, сами значения ещё не вычисляются. Инициализация
default-полей структур, default-экземпляров классов и глобальных переменных откладывается до начала main,
потому что default-выражения могут требовать обычную генерацию expression-кода и могут ссылаться на уже
подготовленные StructName.field или ClassName.field. Только после резервирования этих слотов CodeGen
переходит к полноценному обходу деклараций и компиляции функций.
