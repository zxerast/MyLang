# ТЗ семестрового проекта по разработке языка программирования и его компилятора v0.1

## Описание проекта

Целью проекта является разработка собственного языка программирования
и реализация компилятора для него на языке C++.

Студент самостоятельно проектирует синтаксис и семантику языка,
фиксирует их в спецификации, после чего реализует полный pipeline компиляции:
лексический анализ, синтаксический анализ, семантическую проверку
и генерацию кода для платформы x86-64.

## Требования к языку программирования

### База

Следующие требования являются обязательными для всех студентов.
Отклонение от базовых требований (например, отсутствие отдельных конструкций
или изменение семантики) допускается только по согласованию и должно быть явно отражено в спецификации языка.

#### Лексика

Студент определяет алфавит языка и лексические единицы в `grammar.md`.

Язык обязан поддерживать:
- **Ключевые слова** — зарезервированы и не могут использоваться как идентификаторы
- **Идентификаторы** — имена переменных, функций, типов и т.д.
- **Литералы** — целые и вещественные числа, строки, булевы значения
- **Операторы и разделители**
- **Комментарии** — однострочные; не влияют на семантику программы

#### Синтаксис

Студент определяет полную грамматику языка в `grammar.md` в форме BNF или EBNF.

Грамматика обязана покрывать:
- структуру программы и точку входа (`main`)
- все объявления — переменных, функций, структур, синонимов типов, пространств имён
- все выражения — арифметические, логические, сравнения, индексирование, доступ к полям
- все конструкции управления потоком — блоки, `if/else`, `while`, `break`, `continue`, `return`, нулевая инструкция (инструкция без эффекта)
- литералы массивов и структур

Выражения и инструкции допускаются только внутри тела функций.
На верхнем уровне допускаются только объявления.

#### Семантика

Студент определяет семантику всех конструкций языка в `semantics.md`.

Язык обязан определять:
- **Стратегия вычисления** — call-by-value: аргументы передаются в функцию по значению
- **Семантику присваивания** — value semantics (копирование по значению)
- **Инициализация переменных** — использование неинициализированной переменной
  является ошибкой компиляции
- **Время жизни переменных** — переменная существует в пределах блока, в котором объявлена
- **Побочные эффекты** — функции могут иметь побочные эффекты (ввод/вывод, изменение переменных)
- **Поведение при ошибках времени выполнения** — при возникновении ошибки
  программа завершается с сообщением вида:
```
  runtime error: <сообщение> at line <N>
```
  Обязательно обрабатываются: деление на ноль, выход за границы массива.

#### Типы и система типов

Язык обязан поддерживать следующие встроенные типы:

| Тип | Размер | Описание |
|-----|--------|----------|
| `int8` .. `int64` | 8, 16, 32, 64 бит | знаковое целое число |
| `uint8` .. `uint64` | 8, 16, 32, 64 бит | беззнаковое целое число |
| `float32`, `float64` | 32, 64 бит | вещественное число |
| `bool` | — | булево значение (`true` / `false`) |
| `string` | — | строка |
| `void` / `unit` | — | отсутствие значения |
| `array` | — | массив фиксированного размера, размер является частью типа |
| `struct` | — | пользовательский составной тип |

(Названия типов на усмотрение студента, приведённые выше — пример.)

Система типов обязана соответствовать следующим правилам:
- **Номинальная типизация** — два типа совместимы только если их имена совпадают
- **Отсутствие неявных приведений** — любое преобразование типов должно быть явным
- **Value semantics** — все значения передаются и присваиваются по значению

Поведение при приведениях и переполнении определяется спецификацией языка:
- допустимость widening и narrowing приведений — на усмотрение студента
- поведение при переполнении (overflow) — на усмотрение студента
- тип числовых литералов — на усмотрение студента (суффиксы, автоматическое определение и др.)

**Строки:**
- конкатенация
- получение длины
- сравнение на равенство и неравенство
- реализация строк (UTF-8 или другой способ) — на усмотрение студента

Синонимы типов (type aliases)
- язык обязан поддерживать объявление синонимов типов
- синоним не создаёт новый тип — это альтернативное имя для существующего
- пример: `type Meters = int32`

#### Выражения

Язык обязан поддерживать следующие виды выражений:

**Унарные:** унарный минус, логическое отрицание

**Арифметические:** сложение, вычитание, умножение, деление, остаток от деления

**Логические:** конъюнкция, дизъюнкция, отрицание

**Сравнения:** равенство, неравенство, меньше, больше, меньше или равно, больше или равно

**Приведение типов (cast):** явное приведение одного типа к другому

**Доступ:**
- индексирование массива
- доступ к полю структуры
- доступ к элементу в пространстве имён

**Литералы составных типов:**
- литерал массива
- литерал структуры

**Приоритет и ассоциативность** операторов определяются спецификацией языка.

**Порядок вычисления** операндов определяется спецификацией языка.

> Выбор конкретных символов и ключевых слов для операторов — на усмотрение студента
> и определяется спецификацией языка.

#### Управление потоком

Язык обязан поддерживать следующие конструкции управления потоком:

**Блоки** — последовательность инструкций, вводит новую область видимости;
синтаксис блока (фигурные скобки, отступы, разделители) — на усмотрение студента

**Нулевая инструкция** — пустая инструкция, не производящая никакого эффекта

**Ветвление:**
- `if` / `else` — условное ветвление; ветка `else` не обязательна

**Циклы:**
- `while` — цикл с предусловием
- `break` — немедленное завершение цикла
- `continue` — переход к следующей итерации цикла

**Возврат из функции:**
- `return` — возврат значения из функции
- в функциях с типом возврата `void`/`unit` — `return` без значения

> Выбор конкретных ключевых слов — на усмотрение студента
> и определяется спецификацией языка.

#### Объявления

Язык обязан поддерживать следующие виды объявлений:

**Переменные** — объявление именованного значения с указанием типа и начального значения.
**Мутабельность переменных** — язык должен различать мутабельные и иммутабельные
переменные; конкретный синтаксис (`const`, `let`/`var`, и др.) — на усмотрение студента

**Функции** — объявление именованной функции с параметрами, типом возврата и телом.

**Точка входа** — программа обязана содержать функцию `main` с типом возврата `int`;
возвращаемое значение передаётся как код завершения процесса

**Структуры** — объявление именованного составного типа с полями

**Синонимы типов** — объявление альтернативного имени для существующего типа

**Пространства имён** — объявление именованной области видимости для группировки объявлений

**Модули** — каждый исходный файл является модулем

> Выбор конкретных ключевых слов — на усмотрение студента
> и определяется спецификацией языка.

#### Разрешение имён

Язык использует лексическую область видимости (lexical scoping).

Правила разрешения имён:
- идентификатор разрешается в ближайшей охватывающей области видимости
- перекрытие имён (shadowing) допускается — внутреннее объявление скрывает внешнее
- использование идентификатора до его объявления не допускается
- одно и то же имя не может быть объявлено дважды в одной области видимости

#### Встроенные инструменты

Язык обязан поддерживать следующие встроенные функции:

**Ввод/вывод:**
- `print` — вывод значения в стандартный поток вывода
- `input` — чтение строки из стандартного потока ввода

**Управление выполнением:**
- `exit` — завершение программы с кодом возврата
- `panic` — аварийное завершение программы с сообщением об ошибке

> Конкретный синтаксис вызова и имена функций — на усмотрение студента
> и определяются спецификацией языка.

### Дополнительно

Студент может расширить язык одной или несколькими из следующих возможностей.
Каждая дополнительная возможность должна быть отражена в спецификации языка.

**Система типов:**
- Неявное приведение типов — автоматическое преобразование одного типа к другому в определённых контекстах (например, числовые литералы, арифметические операции)
- Вывод типов — автоматическое определение типа переменной из контекста инициализации, а также типа возвращаемого значения функции из её тела

**Типы данных:**
- Кортежи — фиксированная последовательность значений разных типов
- Динамические массивы (размер определяется во время выполнения)
- ADT (algebraic data types) — объединения/варианты/перечисления с поддержкой pattern matching
- Обобщённые типы (generics / templates)
- Ссылочные типы — переход от value semantics к reference semantics (указатели, ссылки, владение и заимствование)

**Функции:**
- перегрузка функций
- функции как значения (first-class functions) и лямбда-выражения
- методы структур (impls)

**Модульная система:**
- полноценные модули с поддержкой экспорта и импорта (или контроля видимости) объявлений между файлами

## Требования к компилятору

### Архитектура компилятора

#### База

Компилятор должен быть реализован в виде последовательного pipeline из независимых фаз.
Каждая фаза принимает на вход результат предыдущей и не зависит от деталей её реализации.
```
Исходный код (.lang)
       ↓
   [Лексер]          → поток токенов
       ↓
   [Парсер]          → AST
       ↓
  [Семантика]        → аннотированный AST
       ↓
   [Кодоген]          → x86-64 asm / LLVM IR
       ↓
  Исполняемый файл
```

Каждая фаза должна быть реализована отдельным модулем/классом.
Смешивание фаз (например, генерация кода внутри парсера) не допускается.

#### Доп

Между семантическим анализатором и генератором кода вводится явная фаза
линеаризации — промежуточное представление (IR):
```
  [Семантика]        → аннотированный AST
       ↓
   [Lowering]        → IR (трёхадресный код)
       ↓
  [Оптимизатор]      → оптимизированный IR
       ↓
   [Кодоген]         → x86-64 asm / LLVM IR
```

### Лексический анализатор

#### База

Лексический анализатор преобразует исходный текст в поток токенов.

Каждый токен содержит:
- тип
- лексему и/или позицию в исходном коде

Лексер обязан распознавать все лексические единицы согласно спецификации языка:
- ключевые слова
- идентификаторы
- литералы
- операторы и разделители
- комментарии (в поток токенов не передаются)

При обнаружении ошибки лексер обязан выдать диагностическое сообщение
с указанием позиции и прекратить компиляцию.

### Синтаксический анализатор

#### База

Синтаксический анализатор преобразует поток токенов в абстрактное синтаксическое дерево (AST).

Парсер обязан:
- распознавать все синтаксические конструкции согласно спецификации языка
- строить AST, отражающий структуру программы
- выдавать диагностическое сообщение при синтаксической ошибке с указанием позиции
- остановиться на первой синтаксической ошибке

Метод реализации — рекурсивный спуск (Recursive Descent).

Реализация AST — на усмотрение студента, возможные варианты:
- иерархия классов для каждого вида узла
- variant-тип с тегами для разных видов узлов
- другой способ, обеспечивающий удобство построения и обхода дерева

#### Доп

- Восстановление после ошибки — продолжить разбор и накопить несколько ошибок
- Использование субпарсеров для выражений (например, Pratt parsing)
  вместо классического рекурсивного спуска
- Использование более сложного метода разбора (PEG, LR, GLR и др.)
  вместо рекурсивного спуска

### Семантический анализатор

#### База

Семантический анализатор проверяет корректность программы и производит
аннотированный AST, используемый генератором кода.

Анализатор обязан выполнять:
- разрешение имён — все идентификаторы должны быть объявлены до использования
- проверку области видимости — согласно спецификации языка
- проверку типов — согласно системе типов спецификации языка
- проверку корректности управления потоком согласно спецификации языка
- другие семантические проверки, предусмотренные спецификацией языка

### Промежуточное представление и оптимизация (IR) (Доп)

IR должен:
- быть линейным (трёхадресный код или аналог)
- поддерживать все конструкции базового языка
- допускать печать в читаемом текстовом формате

Оптимизации на уровне IR (одна или более на выбор студента):
- constant folding
- dead code elimination
- удаление общих подвыражений (CSE)

### Генератор кода

#### База

Генератор кода транслирует аннотированный AST (или IR, если реализован)
в исполняемый файл для платформы x86-64.

Студент выбирает один из двух подходов:
- **Нативная генерация** — генерация x86-64 ассемблера (NASM, AT&T или аналог)
- **LLVM IR** — генерация LLVM IR с последующей компиляцией через `llc`

Генератор обязан поддерживать все конструкции базового языка
согласно спецификации.

### Интерфейс командной строки

#### База

Компилятор запускается из командной строки:
```
myc <source file> [-o <output file>]
```

Если `-o` не указан — имя выходного файла определяется из имени исходного.

Поддерживаемые флаги:
- `--dump-tokens` — вывести поток токенов и завершить работу
- `--dump-ast` — вывести AST и завершить работу

Компилятор должен возвращать:
- `0` — компиляция успешна
- ненулевой код — при наличии ошибок

### Диагностика ошибок

#### База

Все диагностические сообщения выводятся в `stderr` в едином формате:
```
<файл>:<строка>:<столбец>: error: <сообщение>
```

При наличии ошибок компилятор завершает работу с ненулевым кодом возврата.

## Требования к организации проекта

### Структура проекта

Репозиторий проекта должен содержать:
```
project/
├── src/          # исходный код компилятора
├── specs/        # спецификация языка
├── examples/     # примеры программ на разработанном языке
├── report.md     # отчёт о проделанной работе
├── CMakeLists.txt / Makefile
├── .gitignore
└── README.md
```

### Система сборки

Проект должен собираться одной командой из корня репозитория.

Поддерживаемые системы сборки:
- **CMake** (предпочтительно)
- **Makefile**

Требования:
- стандарт: **C++23**
- сборка должна проходить без ошибок
- поддерживаемые компиляторы: `clang++`, `g++`, `MSVC`
- обязательные цели: `build`, `run`, `debug`, `clean`

### Система контроля версий

Проект обязан вестись в Git-репозитории.

Требования:
- история коммитов должна отражать процесс разработки
- коммиты делаются регулярно — не допускается один коммит с готовым проектом
- репозиторий размещается на GitHub / GitLab

### Спецификация языка

Спецификация языка оформляется в виде набора документов в папке `specs/`:

- `grammar.md` — лексическая и синтаксическая грамматика (BNF или EBNF)
- `semantics.md` — семантика всех конструкций языка
- `types.md` — система типов
- `codegen.md` — описание генератора кода: выбор платформы (нативный x86-64 или LLVM IR),
  соглашения о вызовах, схема распределения регистров/памяти, а также описание IR и оптимизаций, если они реализованы

Спецификация должна быть достаточно полной, чтобы по ней можно было
реализовать компилятор независимо от исходного кода.

### Отчёт о работе

Отчёт оформляется в виде документа `report.md` в корне репозитория и содержит:

- описание архитектуры компилятора и принятых проектных решений
- обоснование выбора подходов (метод разбора, платформа генерации кода и др.)
- известные ограничения и нереализованные возможности
- инструкцию по сборке и запуску
- описание реализованных дополнительных возможностей (если есть)

## Требования к коду на C++

### Стандарт

Компилятор реализуется на **C++23**. Поддерживаемые компиляторы: `clang++`, `g++`, `MSVC`.

### Стиль и качество кода

- код должен быть единообразно оформлен на протяжении всего проекта
- каждый модуль/класс должен иметь краткий комментарий с описанием назначения
- использование «магических чисел» и необъяснимых конструкций без комментариев не допускается

### Ограничения

Запрещено:
- использование генераторов лексеров и парсеров (`flex`, `bison`, `antlr` и аналоги)
- использование сторонних библиотек, не согласованных с преподавателем

Разрешено:
- стандартная библиотека C++23
- регулярные выражения (`std::regex`)
- LLVM (если выбрана соответствующая платформа генерации кода)
- другие библиотеки — по согласованию с преподавателем

### Обработка ошибок

Для передачи диагностических ошибок компилятора (лексических, синтаксических,
семантических) используется `std::expected` или `std::optional`.

Использование кодов возврата для обозначения ошибок не рекомендуется.

Исключения (`throw`) допускаются только для критических внутренних ошибок
компилятора (например, недостаток памяти, недостижимые ветки кода).

Использование исключений для управления потоком диагностики не допускается.

Также не допускается использование глобальных переменных для хранения состояния ошибок.
